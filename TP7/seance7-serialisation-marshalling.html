3. Cours — Concepts fondamentaux (distribué)

3.1 Définitions clés

TermeDéfinitionAnalogie
Sérialisation Transformer une structure de données en mémoire (objet, dictionnaire…) en une séquence d'octets (ou de caractères) transportable ou stockable. Emballer un meuble IKEA dans un carton plat pour le livrer.
Désérialisation Opération inverse : reconstruire la structure de données à partir de la séquence d'octets. Remonter le meuble à partir des planches et du plan.
Marshalling Sérialisation + métadonnées nécessaires au transport (types, longueurs, en-têtes de protocole). Souvent utilisé dans les contextes RPC. Emballer + coller l'étiquette d'expédition avec poids, dimensions, destinataire.
Unmarshalling Désérialisation + interprétation des métadonnées de transport. Recevoir le colis, lire l'étiquette, puis déballer.

3.2 Pourquoi c'est central en systèmes distribués

La sérialisation intervient partout dans une architecture distribuée :

  • Échanges réseau — requêtes/réponses HTTP, messages gRPC, WebSocket
  • Stockage — bases de données documentaires, fichiers de configuration
  • Caches — Redis, Memcached : les objets doivent être sérialisés en bytes
  • Logs structurés — chaque événement est sérialisé (JSON, etc.)
  • Files de messages — Kafka, RabbitMQ : les messages sont des bytes
  • RPC (Remote Procedure Call) — gRPC marshalle/unmarshalle automatiquement

3.3 Le contrat de données

Même en JSON (format sans schéma formel obligatoire), il existe un contrat implicite entre émetteur et récepteur :

  • Quels champs sont attendus ?
  • Quels types pour chaque champ ?
  • Quels champs sont obligatoires vs optionnels ?
  • Que se passe-t-il si un champ inconnu apparaît ? (tolérer ou refuser ?)
  • Comment gérer le versioning (évolution du contrat) ?
⚠️ Piège classique : confondre format (JSON) et contrat (schéma attendu)

« C'est du JSON, donc c'est compatible. » — FAUX.
JSON n'est qu'un format de représentation textuelle. Deux services peuvent parler JSON mais avoir des contrats incompatibles : un attend "status": "active" (chaîne) et l'autre envoie "status": 1 (entier).
Le format est le même, le contrat est différent → erreur silencieuse ou crash.

4. Cours — Formats : texte vs binaire, schema-less vs schema-based

4.1 Deux grandes familles

Formats texte (JSON, XML, YAML) : représentation lisible par l'humain, encodée en caractères (UTF-8 généralement). Plus volumineux, mais faciles à débugger.

Formats binaires (Protocol Buffers, MessagePack, CBOR, Avro) : représentation compacte en octets, illisible directement. Plus performants et plus petits.

4.2 Schema-less vs schema-based

Schema-less : le format ne force pas de schéma formel. JSON est schema-less par nature — l'émetteur peut envoyer n'importe quelle structure. La validation se fait côté applicatif.

Schema-based : le format impose un schéma prédéfini (fichier .proto, .avsc…). Le code de sérialisation/désérialisation est généré à partir du schéma. Validation intrinsèque.

Figure 2 — Schema-less vs Schema-based : compromis fondamentaux.

Il n'y a pas de « meilleur format universel ». Le choix dépend du contexte : JSON pour l'interopérabilité et le Web, Protobuf pour la performance et le versioning strict, XML quand l'écosystème l'impose (legacy, standards industriels). En cybersécurité, le format influence directement la surface d'attaque.

⚠️ Piège classique : les flottants en JSON

json.loads('{"price": 19.99}')["price"] donne 19.99 en Python, mais c'est un float IEEE 754. Pour des montants financiers, préférer transmettre en centimes (entier) ou en string et convertir avec decimal.Decimal côté récepteur.

5.2 Sérialiser / désérialiser : json.dumps & json.loads

import json

# --- Sérialisation ---
data = {"id": 42, "title": "Rapport Q1", "tags": ["finance", "interne"]}

# Bonne pratique : ensure_ascii=False pour garder les accents lisibles
json_str = json.dumps(data, ensure_ascii=False, indent=2)
print(json_str)
# {
#   "id": 42,
#   "title": "Rapport Q1",
#   "tags": ["finance", "interne"]
# }

# --- Sérialisation en bytes (pour envoi réseau) ---
json_bytes = json_str.encode("utf-8")

# --- Désérialisation ---
received = json.loads(json_bytes)  # accepte str ou bytes
print(received["title"])  # Rapport Q1

5.4 Validation « sans librairie externe »

En production, on utilise souvent des librairies de validation (pydantic, marshmallow…). Mais il est essentiel de comprendre la validation manuelle pour savoir ce qui se passe « sous le capot » :

def validate_document(data: dict) -> dict:
    """Valide et nettoie un payload Document. Lève ValueError si invalide."""
    errors = []

    # Présence des champs obligatoires
    for field in ("id", "title", "author"):
        if field not in data:
            errors.append(f"Champ obligatoire manquant : {field}")

    # Type checking
    if "id" in data and not isinstance(data["id"], int):
        errors.append("'id' doit être un entier")

    if "title" in data:
        if not isinstance(data["title"], str):
            errors.append("'title' doit être une chaîne")
        elif len(data["title"]) > 200:
            errors.append("'title' trop long (max 200 caractères)")
        elif len(data["title"].strip()) == 0:
            errors.append("'title' ne peut être vide")

    # Valeurs autorisées (allowlist)
    ALLOWED_CLASSIFICATIONS = {"public", "internal", "confidential", "secret"}
    classification = data.get("classification", "internal")
    if classification not in ALLOWED_CLASSIFICATIONS:
        errors.append(f"'classification' invalide : {classification}")

    if errors:
        raise ValueError(f"Validation échouée : {'; '.join(errors)}")

    return data

5.5 Stratégies de compatibilité (versioning JSON)

  • Champs optionnels : tout nouveau champ doit avoir une valeur par défaut côté récepteur.
  • Tolérance à l'inconnu : ignorer les champs non reconnus (ne pas planter) — mais ne pas les stocker aveuglément.
  • Ne jamais retirer un champ obligatoire sans plan de migration.
  • Ne jamais changer le type d'un champ existant (string → int = cassure).
🔒 Point cybersécurité : validation d'entrée ≠ confiance

Le fait de valider un JSON (syntaxe correcte, champs présents, types OK) ne signifie pas que les données sont fiables ou bienveillantes. La validation est un filtre nécessaire mais non suffisant. Il faut aussi :

  • Vérifier l'origine des données (authentification, signature)
  • Limiter les valeurs (allowlist, bornes numériques, longueurs max)
  • Appliquer le principe du moindre privilège : n'accepter que ce qui est strictement nécessaire
  • Approche « fail closed » : en cas de doute, refuser plutôt qu'accepter

6. Cours — XML en Python (usage + sécurité) XML Sécurité

6.1 Pourquoi XML existe encore

  • Legacy : des milliers de systèmes (SOAP, WS-*, bancaire, santé) reposent sur XML.
  • Configuration : pom.xml (Maven), manifestes Android, fichiers Office (OOXML).
  • Standards industriels : HL7 (santé), XBRL (finance), SVG (graphiques).
  • Schémas riches : XSD permet des validations très précises (expressions régulières sur les valeurs, cardinalité, héritage de types).

Même si JSON domine les nouvelles API, un ingénieur cybersécurité doit connaître XML — ne serait-ce que pour défendre les systèmes existants.

6.2 Parsing Python : choix et pièges

La bibliothèque standard Python offre plusieurs parseurs XML :

ModuleTypeSécurité par défautUsage recommandé
xml.etree.ElementTree Arbre (DOM-like) ⚠️ Vulnérable XXE par défaut (versions anciennes) Usage interne, données fiables uniquement
xml.sax Événementiel (SAX) ⚠️ Configurable Gros fichiers, streaming
xml.dom.minidom DOM complet ⚠️ Lourd, vulnérable Déconseillé en production
defusedxml (tiers) Wrapper sécurisé ✅ Désactive les fonctionnalités dangereuses Entrées non fiables

6.3 Risques XML à connaître (défensif)

XXE — XML External Entity

Concept : XML permet de définir des « entités » qui référencent des ressources externes (fichiers locaux, URLs). Un attaquant peut inclure dans son payload XML une entité qui pointe vers /etc/passwd ou une URL interne. Si le parseur résout cette entité, il exfiltre le contenu vers l'attaquant.

Impact : lecture de fichiers serveur, SSRF (Server-Side Request Forgery), déni de service.

Bombe XML (Billion Laughs)

Concept : un XML de quelques Ko qui définit des entités se référençant entre elles de manière exponentielle. Le parseur tente de les résoudre → consommation mémoire/CPU explosive → déni de service.

6.4 Mesures de protection

  • Éviter XML si inutile : pour une nouvelle API, préférer JSON ou Protobuf.
  • Limiter la taille du payload avant même de parser (ex : max 1 Mo).
  • Parseur sûr : utiliser defusedxml qui désactive par défaut les entités externes, la résolution DTD, les entités récursives.
  • Désactiver les features dangereuses : pas de résolution d'entités externes, pas de DTD processing, pas de XInclude.
  • Valider le contenu après parsing : mêmes principes que JSON (présence, types, longueurs, allowlist).

7. Cours — Protocol Buffers (Protobuf) côté Python Protobuf

7.1 L'idée fondamentale

Protocol Buffers (développé par Google) est un format de sérialisation binaire, compact, rapide, et — surtout — basé sur un schéma formel. Le workflow :

  • Décrire les données dans un fichier .proto
  • Compiler ce fichier avec protoc pour générer du code Python
  • Utiliser le code généré pour encoder/décoder
Figure 4 — Protobuf : .proto → génération de code → encode/decode.

7.2 Exemple de fichier .proto

// document.proto
syntax = "proto3";

message Document {
    int32  id     = 1;          // Numéro de champ unique
    string title  = 2;
    string author = 3;
    repeated string tags = 4;  // Liste (repeated = 0..N)
}

7.3 Notions cruciales pour le versioning

RègleExplicationConséquence si violée
Ne jamais réutiliser un numéro de champ Un numéro = un champ pour toujours. Si on supprime le champ 3, on le marque reserved. Données corrompues silencieusement
Ajouter des champs optionnels Les nouveaux champs ont une valeur par défaut (0, "", false). Les anciens lecteurs les ignorent. Rétro/antéro-compatible ✅
Ne jamais changer le type d'un champ int32 → string = cassure binaire. Décodage erroné, crash
Ne jamais changer le numéro d'un champ Le numéro est l'identifiant binaire. Données mappées au mauvais champ

Ajout de champ vs changement de type

// Version 1
message Document {
    int32  id     = 1;
    string title  = 2;
    string author = 3;
}

// Version 2 — COMPATIBLE ✅ (ajout de champs optionnels)
message Document {
    int32  id             = 1;
    string title          = 2;
    string author         = 3;
    repeated string tags  = 4;  // NOUVEAU — ignoré par v1
    string classification  = 5;  // NOUVEAU — défaut ""
}

// Version CASSÉE ❌ (changement de type du champ 1)
message Document {
    string id = 1;  // ❌ Était int32 → CASSURE BINAIRE
    ...
}
📌 À retenir — Protobuf en 4 points
  • Le schéma (.proto) est le contrat formel entre services.
  • Les numéros de champ sont immuables : on ne les change ni ne les réutilise.
  • Ajouter un champ optionnel est toujours safe.
  • Installer la librairie Python : pip install protobuf ; compiler avec protoc --python_out=. document.proto.

8. Cours — Désérialisation non sécurisée : comprendre et se protéger Sécurité

8.1 Le problème fondamental

La désérialisation, c'est transformer des bytes en objets vivants dans la mémoire du programme. Si l'entrée est contrôlée par un attaquant, cette transformation peut déclencher des comportements inattendus :

  • Exécution de code arbitraire (si le format permet de représenter des objets exécutables)
  • Instanciation d'objets dangereux
  • Déni de service (payload géant, récursion infinie)
  • Fuite d'information (erreurs détaillées, sérialisation de champs internes)

Analogie : Imaginez recevoir un colis postal qui, une fois ouvert, exécute automatiquement les instructions écrites dessus. C'est exactement ce que fait pickle.loads() avec des données non fiables.

8.2 Côté Python : pourquoi pickle est dangereux avec entrée non fiable

Le module pickle de Python permet de sérialiser n'importe quel objet Python, y compris des objets qui, lors de leur reconstruction, exécutent du code arbitraire. Le protocole pickle contient littéralement des instructions de machine virtuelle qui sont interprétées lors du loads().

⚠️ Piège classique : « pickle c'est pratique, je l'utilise partout »

pickle est acceptable dans un contexte de confiance totale :

  • Cache local entre processus du même serveur
  • Stockage temporaire de données que vous seul avez produites
  • Communication inter-processus sur la même machine, avec données signées (HMAC)

pickle est interdit pour :

  • Toute donnée reçue du réseau (API, WebSocket, fichier uploadé…)
  • Toute donnée stockée dans un emplacement modifiable par un tiers
  • Tout échange entre services distincts (même internes)

8.3 Autres surfaces de risque

eval() / exec() sur données désérialisées Exécution de code arbitraire Ne jamais utiliser sur des entrées utilisateur

🔒 Point cybersécurité : la règle d'or

Ne jamais désérialiser des données non fiables dans un format qui permet l'exécution de code.
En Python : pickle, yaml.load(), eval(), exec() sont des « formats exécutables ». json.loads() et Protobuf ParseFromString() sont des formats inertes (pas d'exécution de code lors du décodage).

9. Démonstrations / Live Coding Python

9.1 Démo 1 — JSON robuste (stdlib) JSON Validation

import json
from dataclasses import dataclass, field, asdict
from typing import List, Optional
import logging

logger = logging.getLogger(__name__)

# ──────────────────────────────────────────────
# MODÈLE : Document (via dataclass)
# ──────────────────────────────────────────────
@dataclass
class Document:
    id: int
    title: str
    author: str
    tags: List[str] = field(default_factory=list)
    classification: str = "internal"
    # Champ sensible : ne sera PAS sérialisé
    _internal_score: float = field(default=0.0, repr=False)


# ──────────────────────────────────────────────
# SÉRIALISATION : exclure les champs sensibles
# ──────────────────────────────────────────────
EXCLUDED_FIELDS = {"_internal_score"}

def serialize_document(doc: Document) -> str:
    """Sérialise un Document en JSON, sans champs sensibles."""
    data = {k: v for k, v in asdict(doc).items()
            if k not in EXCLUDED_FIELDS}
    return json.dumps(data, ensure_ascii=False)


# ──────────────────────────────────────────────
# VALIDATION + DÉSÉRIALISATION
# ──────────────────────────────────────────────
ALLOWED_CLASSIFICATIONS = {"public", "internal", "confidential", "secret"}
MAX_TITLE_LEN = 200
MAX_TAGS = 20

def deserialize_document(raw: str) -> Document:
    """Désérialise et valide un JSON en Document."""
    # Étape 1 : parsing JSON
    try:
        data = json.loads(raw)
    except json.JSONDecodeError as e:
        logger.warning("JSON invalide : %s", e)
        raise ValueError("Payload invalide")

    # Étape 2 : doit être un dict
    if not isinstance(data, dict):
        raise ValueError("Payload invalide")

    errors = []

    # Étape 3 : champs obligatoires
    for f in ("id", "title", "author"):
        if f not in data:
            errors.append(f"Champ obligatoire manquant : {f}")

    # Étape 4 : vérification de types
    if "id" in data and not isinstance(data["id"], int):
        errors.append("'id' doit être un entier")

    if "title" in data:
        if not isinstance(data["title"], str):
            errors.append("'title' doit être une chaîne")
        elif len(data["title"]) > MAX_TITLE_LEN:
            errors.append("Titre trop long")

    if "author" in data and not isinstance(data["author"], str):
        errors.append("'author' doit être une chaîne")

    # Étape 5 : tags (optionnel, liste de strings)
    tags = data.get("tags", [])
    if not isinstance(tags, list):
        errors.append("'tags' doit être une liste")
    elif len(tags) > MAX_TAGS:
        errors.append("Trop de tags")
    elif not all(isinstance(t, str) for t in tags):
        errors.append("Chaque tag doit être une chaîne")

    # Étape 6 : classification (optionnel, allowlist)
    classification = data.get("classification", "internal")
    if classification not in ALLOWED_CLASSIFICATIONS:
        errors.append("Classification non autorisée")

    # Étape 7 : fail closed
    if errors:
        logger.warning("Validation échouée : %s", errors)
        raise ValueError("Payload invalide")  # Message générique côté client

    return Document(
        id=data["id"],
        title=data["title"].strip(),
        author=data["author"].strip(),
        tags=tags,
        classification=classification,
    )

9.2 Démo 2 — XML parsing (stdlib) + rappel sécurité XML Sécurité

import xml.etree.ElementTree as ET
import logging

logger = logging.getLogger(__name__)

# ──────────────────────────────────────────────
# CONSTANTES DE SÉCURITÉ
# ──────────────────────────────────────────────
MAX_XML_SIZE = 1_000_000  # 1 Mo maximum
EXPECTED_FIELDS = {"id", "title", "author", "date"}


def parse_document_xml(raw_xml: str) -> dict:
    """Parse un XML de métadonnées document de manière défensive."""

    # Contrôle 1 : taille maximale AVANT tout parsing
    if len(raw_xml) > MAX_XML_SIZE:
        logger.warning("XML trop volumineux : %d octets", len(raw_xml))
        raise ValueError("Payload trop volumineux")

    # Contrôle 2 : rejet basique de DOCTYPE (défense en profondeur)
    if " in raw_xml.upper() or " in raw_xml.upper():
        logger.warning("DOCTYPE/ENTITY détecté dans XML — rejeté")
        raise ValueError("Payload invalide")

    # Contrôle 3 : parsing
    try:
        root = ET.fromstring(raw_xml)
    except ET.ParseError as e:
        logger.warning("XML mal formé : %s", e)
        raise ValueError("Payload invalide")

    # Contrôle 4 : extraire uniquement les champs attendus
    result = {}
    for field_name in EXPECTED_FIELDS:
        elem = root.find(field_name)
        if elem is not None and elem.text:
            result[field_name] = elem.text.strip()

    # Contrôle 5 : champs obligatoires
    missing = {"id", "title"} - result.keys()
    if missing:
        logger.warning("Champs manquants dans XML : %s", missing)
        raise ValueError("Payload invalide")

    return result


# ──────────────────────────────────────────────
# TEST
# ──────────────────────────────────────────────
if __name__ == "__main__":
    xml_input = """<?xml version="1.0" encoding="UTF-8"?>
    <document>
        <id>42</id>
        <title>Rapport annuel</title>
        <author>Bob</author>
        <date>2026-01-15</date>
    </document>"""

    result = parse_document_xml(xml_input)
    print("Parsé :", result)
🔒 Ce qu'il faut éviter en XML (concept, sans code d'attaque)
  • Ne pas parser d'XML non fiable sans protections : les parseurs par défaut peuvent résoudre des entités externes, ouvrant la voie à l'exfiltration de fichiers.
  • En production avec données non fiables : utiliser defusedxml.ElementTree au lieu de xml.etree.ElementTree. Le module defusedxml désactive automatiquement les fonctionnalités dangereuses.
  • Toujours limiter la taille avant parsing — un XML de 10 Ko peut consommer des Go de RAM si des entités sont imbriquées.

11.3 Concept HMAC (intégrité)

HMAC (Hash-based Message Authentication Code) : on calcule un hash du payload avec une clé secrète partagée entre émetteur et récepteur. Le récepteur recalcule le HMAC et compare — si différent, le payload a été altéré.

import hmac
import hashlib
import json

SECRET_KEY = b"cle_partagee_entre_services"  # En prod : variable d'env, vault

def sign_payload(data: dict) -> tuple:
    """Retourne (payload_json, signature_hex)."""
    payload = json.dumps(data, sort_keys=True, ensure_ascii=False)
    sig = hmac.new(SECRET_KEY, payload.encode("utf-8"), hashlib.sha256)
    return payload, sig.hexdigest()

def verify_payload(payload_json: str, received_sig: str) -> bool:
    """Vérifie l'intégrité du payload."""
    expected = hmac.new(SECRET_KEY, payload_json.encode("utf-8"), hashlib.sha256)
    return hmac.compare_digest(expected.hexdigest(), received_sig)

11.4 Checklist « Secure Deserialization Baseline »

  • Utiliser uniquement des formats inertes (JSON, Protobuf) pour les entrées non fiables
  • Interdire pickle, yaml.load(), eval(), exec() sur entrées réseau
  • Limiter la taille du payload avant parsing (Content-Length, len())
  • Valider la structure : type racine (dict/list), champs attendus (allowlist)
  • Valider les types de chaque champ (isinstance, pas de coercition implicite)
  • Valider les valeurs : bornes numériques, longueurs, allowlist pour enums
  • Signer les payloads inter-services avec HMAC ou vérifier via TLS mutuel
  • Vérifier l'authentification/autorisation avant la désérialisation
  • Logger les échecs (source, timestamp, type d'erreur) sans données sensibles
  • Appliquer un timeout au parsing pour prévenir les DoS
  • Rejeter par défaut (fail closed) tout payload non conforme
  • Tester régulièrement avec des payloads malformés (fuzzing basique)

13. Mini-quiz (10 questions)

Q1. Quelle est la différence fondamentale entre sérialisation et marshalling ?
Q2. JSON possède-t-il un type natif pour les dates ? Si non, quelle convention est recommandée ?
Q3. Pourquoi dit-on que JSON est « schema-less » ? Cela signifie-t-il qu'il n'y a aucun contrat entre émetteur et récepteur ?
Q4. Quel est le risque principal de pickle.loads() sur des données reçues du réseau ?
Q5. En Protobuf, que se passe-t-il si un lecteur v1 reçoit un message v2 contenant un champ supplémentaire qu'il ne connaît pas ?
Q6. Qu'est-ce que le pattern « fail closed » en validation de données ?
Q7. Citez deux risques spécifiques au parsing XML que JSON ne présente pas.
Q8. Pourquoi ne faut-il jamais réutiliser un numéro de champ en Protobuf, même après suppression ?
Q9. En Python, quelle est la différence entre yaml.load() et yaml.safe_load() du point de vue sécurité ?
Q10. Vous recevez un payload JSON d'un client. Votre validation accepte la syntaxe et les types. Pouvez-vous considérer les données comme fiables ? Justifiez.

Corrigé

R1. La sérialisation transforme un objet en séquence d'octets. Le marshalling inclut en plus les métadonnées de transport nécessaires (types, longueurs, en-têtes) pour que le récepteur puisse correctement interpréter et reconstruire la structure. Le marshalling est un sur-ensemble de la sérialisation, typiquement utilisé en RPC.
R2. Non, JSON n'a pas de type date natif. La convention recommandée est d'utiliser une chaîne au format ISO 8601 (ex : "2026-01-15T10:30:00Z"). Le récepteur la parse explicitement avec datetime.fromisoformat() ou équivalent.
R3. JSON est « schema-less » car le format lui-même n'impose aucune structure prédéfinie — n'importe quelle combinaison de clés/valeurs est syntaxiquement valide. Cependant, il existe toujours un contrat implicite entre émetteur et récepteur (quels champs, quels types, quelles valeurs). Ce contrat doit être défini et validé par le code applicatif (ou via JSON Schema).
R4. pickle.loads() peut exécuter du code arbitraire lors de la reconstruction des objets. Le protocole pickle contient des instructions de machine virtuelle qui sont interprétées pendant le chargement. Un attaquant peut forger un payload pickle qui exécute des commandes système sur le serveur.
R5. Le lecteur v1 ignore silencieusement les champs inconnus. Protobuf est conçu pour la rétrocompatibilité : les champs non reconnus sont simplement sautés lors du décodage, sans erreur. C'est un avantage majeur pour le versioning progressif.
R6. « Fail closed » signifie qu'en cas de doute ou d'ambiguïté lors de la validation, le système refuse l'entrée plutôt que de l'accepter. C'est l'opposé de « fail open » (accepter par défaut). En sécurité, on préfère toujours fail closed : un faux positif (rejet d'un légitime) est préférable à un faux négatif (acceptation d'un malveillant).
R7. XXE (XML External Entity) : résolution d'entités qui pointent vers des fichiers locaux ou des URLs, permettant l'exfiltration de données ou le SSRF. Bombe XML (Billion Laughs) : entités imbriquées exponentiellement causant un déni de service par épuisement mémoire/CPU. JSON ne supporte ni entités ni DTD, donc n'est pas vulnérable à ces attaques.
R8. En Protobuf, le numéro de champ est l'identifiant binaire dans le flux encodé. Si on réutilise le numéro 3 pour un nouveau champ de type différent, les anciens messages (qui avaient l'ancien champ 3) seront décodés incorrectement — les bytes seront interprétés avec le mauvais type, causant des données corrompues silencieusement. Il faut utiliser reserved pour marquer les numéros retirés.
R9. yaml.load() sans spécifier Loader=SafeLoader utilise le loader complet qui peut instancier n'importe quel objet Python via les tags YAML (ex : !!python/object/apply:os.system). Cela permet l'exécution de code arbitraire. yaml.safe_load() (ou Loader=SafeLoader) restreint aux types de base (str, int, list, dict…) et ne permet aucune instanciation d'objet.
R10. Non. La validation syntaxique et de types est une condition nécessaire mais non suffisante. Des données peuvent être syntaxiquement correctes et typées correctement, mais contenir des valeurs malveillantes (injection SQL dans un champ texte, classification élevée non autorisée pour cet utilisateur, etc.). Il faut aussi vérifier l'origine (authentification), les autorisations (l'utilisateur a-t-il le droit d'envoyer ces données ?), les valeurs métier (allowlist, bornes), et idéalement l'intégrité (signature).
