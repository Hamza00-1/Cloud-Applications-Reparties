1. Introduction générale

Dans votre environnement de travail, vous disposez déjà d'une infrastructure virtualisée mature : des hôtes ESXi, un vCenter qui les administre de manière centralisée, et Veeam qui assure les sauvegardes. Vous savez créer des machines virtuelles, les cloner, les migrer à chaud entre hôtes, les sauvegarder. C'est un socle solide, et ce socle reste pleinement valide.

Alors pourquoi parler de Docker et de Kubernetes ? Parce que la virtualisation classique, aussi puissante soit-elle, répond à un certain type de besoin — essentiellement faire tourner plusieurs systèmes d'exploitation sur un même matériel. Or, les applications modernes posent d'autres problèmes : comment livrer une application identique du poste du développeur au serveur de production ? Comment démarrer vingt instances d'un service en quelques secondes pour absorber un pic de charge ? Comment mettre à jour sans interruption de service ? Comment orchestrer des dizaines, voire des centaines de composants applicatifs qui doivent communiquer entre eux ?

C'est à ces questions que répondent successivement la conteneurisation (Docker) et l'orchestration de conteneurs (Kubernetes). La séance d'aujourd'hui pose la première brique : comprendre ce qu'est un conteneur, comment on le construit, comment on l'exécute, et pourquoi on aura très vite besoin d'un outil pour les orchestrer à grande échelle.

Au-delà de Kubernetes, cette chaîne technologique ouvre la porte à tout l'écosystème du MLOps : Kubeflow, MLflow, TensorFlow Serving, pipelines de données reproductibles. Tous reposent fondamentalement sur la conteneurisation. Docker est donc une compétence transverse : une fois maîtrisée, elle irrigue l'ensemble de vos futurs projets cloud et data.

Docker n'est pas un remplaçant de la virtualisation VMware. C'est une couche complémentaire qui s'installe généralement dans une VM Linux hébergée sur votre infrastructure ESXi. Ces deux mondes cohabitent parfaitement. Vous ne jetez rien, vous ajoutez un outil de plus à votre coffre.

2. Partie théorique : comprendre avant de manipuler

Avant de lancer la moindre commande, nous allons poser les fondations conceptuelles. Cette partie est essentielle : un étudiant qui sait taper docker run sans comprendre ce qui se passe au niveau du noyau Linux ne pourra pas dépanner un incident en production, ni concevoir une architecture propre. Le temps investi ici est rentabilisé mille fois lors du TP et surtout lors des séances Kubernetes à venir.

2.1 Pourquoi introduire Docker maintenant ?

Vous savez déployer une application sur une VM. Pourquoi changer une méthode qui fonctionne ? La réponse tient en une observation simple : la VM optimise le matériel, pas le déploiement applicatif. Elle a été pensée dans les années 2000 pour faire tourner plusieurs OS sur un même serveur physique. Mais une fois l'OS en place, vous devez encore y installer l'application, ses dépendances, configurer le réseau, les services, les droits… et refaire tout cela à l'identique sur chaque environnement (dev, test, préprod, prod). C'est là que les ennuis commencent.

Les limites concrètes d'un fonctionnement uniquement en VM

Prenons un exemple que vous connaissez tous. Un développeur termine une application Python qui fonctionne parfaitement sur sa machine. Il l'envoie à l'équipe d'exploitation. Celle-ci la déploie sur une VM de préprod. Surprise : l'application ne démarre pas. Version de Python différente. Bibliothèque manquante. Variable d'environnement absente. Permissions incorrectes sur un répertoire. Le fameux « ça marche chez moi » naît de cette fragilité.

Ce problème n'est pas théorique. Il coûte du temps, de l'argent, et génère des tensions entre équipes. Docker a été conçu pour y répondre en proposant un principe simple : emballer l'application ET son environnement dans un paquet autonome, identique partout.

Voici les limites précises qui motivent l'adoption de Docker :

  • Démarrage lent. Une VM Linux met typiquement 30 à 60 secondes à démarrer. Un conteneur démarre en moins d'une seconde. Quand on doit lancer 50 instances pour absorber un pic, la différence est décisive.
  • Empreinte disque et mémoire. Une VM embarque un OS complet : plusieurs gigaoctets. Dix VM = dix OS. Un conteneur réutilise le noyau de l'hôte : quelques dizaines de mégaoctets suffisent souvent.
  • Reproductibilité fragile. Deux VM installées à six mois d'intervalle avec la même procédure ne sont presque jamais strictement identiques (mises à jour, versions légèrement différentes, patches). Une image Docker, elle, produit toujours le même conteneur.
  • Portabilité limitée. Déplacer une VM d'un hyperviseur à un autre est possible mais fastidieux (formats différents, drivers). Une image Docker tourne à l'identique sur un laptop, un serveur ESXi, un cluster Kubernetes, AWS, Azure…
  • Couplage fort entre application et OS. Dans une VM, l'application partage l'OS avec d'autres choses (monitoring, agents, autres services). Un conteneur, lui, est dédié à un et un seul processus applicatif.
  • Déploiement et mise à jour complexes. Mettre à jour une application sur une VM implique souvent une interruption, un risque de régression, un retour arrière complexe. Avec Docker, on remplace simplement l'image par une nouvelle version.
  • Standardisation absente. Chaque entreprise, chaque équipe réinvente ses propres procédures de packaging. Docker impose un format standard que tout l'écosystème comprend.
Le vrai apport de Docker en une phrase

Docker permet de packager une application avec tout son environnement d'exécution dans une unité légère, portable et reproductible, qui démarre en une fraction de seconde et se comporte identiquement partout où elle est exécutée.

Ces propriétés ne sont pas des bonus esthétiques : elles sont précisément ce qui rend possible l'orchestration à grande échelle. Sans conteneurs, pas de Kubernetes. Sans Kubernetes, pas de MLOps moderne. C'est pour cela que cette séance est un pivot.

2.2 Machine virtuelle vs conteneur : la comparaison fondamentale

Cette section est probablement la plus importante de la partie théorique. Beaucoup d'étudiants pensent qu'un conteneur est « une VM légère ». C'est faux, et cette incompréhension est source de nombreuses erreurs de conception par la suite. Un conteneur et une VM reposent sur des mécanismes fondamentalement différents.

Architecture d'une machine virtuelle

Une VM s'appuie sur un hyperviseur, comme ESXi dans votre environnement. L'hyperviseur émule du matériel virtuel (CPU, RAM, disque, carte réseau) au-dessus du matériel physique. Sur ce matériel virtuel, on installe un système d'exploitation complet, avec son propre noyau, ses propres pilotes, ses propres services système. Chaque VM est donc une machine à part entière, isolée des autres non seulement au niveau applicatif, mais jusqu'à la racine de son OS.

Cette isolation profonde est un avantage (forte sécurité, possibilité de faire cohabiter Linux et Windows sur le même hôte) mais aussi un coût : on multiplie les OS, avec la RAM, le CPU et l'espace disque que cela implique.

Architecture d'un conteneur

Un conteneur, lui, ne virtualise pas le matériel. Il s'appuie sur des fonctionnalités du noyau Linux : les namespaces (qui isolent la vue des processus, du réseau, du système de fichiers) et les cgroups (qui limitent les ressources consommées). Concrètement, un conteneur est un ou plusieurs processus qui tournent sur l'OS hôte, mais qui croient être seuls au monde grâce à ces mécanismes d'isolation.

Il n'y a donc qu'un seul noyau, celui de l'hôte, partagé par tous les conteneurs. C'est ce partage qui explique la légèreté et la vitesse de démarrage : pas besoin d'initialiser un second noyau, pas besoin de boot complet, pas besoin de services système redondants.

Les acteurs en jeu : une clarification nécessaire

Quand on parle de VM et de conteneur, plusieurs éléments interviennent. Clarifions leurs rôles respectifs, car la confusion est fréquente.

🖥️ La VM : pour quoi faire ?

Points forts : isolation très forte, possibilité de faire tourner des OS différents, longévité, maturité de l'écosystème (sauvegarde, migration à chaud, snapshots).

Limites : lourdeur, lenteur de démarrage, empreinte disque importante, procédures de déploiement hétérogènes entre équipes.

À privilégier quand : on doit isoler complètement un OS, héberger du legacy Windows, respecter des contraintes de sécurité très fortes au niveau OS.

📦 Le conteneur : pour quoi faire ?

Points forts : démarrage immédiat, faible empreinte, portabilité totale, reproductibilité garantie, scalabilité horizontale facile.

Limites : noyau partagé (moindre isolation), complexité à gérer à grande échelle sans orchestrateur, applications stateful plus délicates.

À privilégier quand : on veut packager une application, déployer rapidement, scaler dynamiquement, construire une chaîne CI/CD moderne, préparer le MLOps.

Confusion classique à éviter

« Docker remplace VMware. » Faux. Les deux technologies répondent à des problèmes différents et se combinent très bien. Dans la réalité, une architecture moderne typique ressemble à ceci : des hôtes ESXi → des VM Linux → Docker installé dans ces VM → conteneurs applicatifs. Plus tard : Kubernetes orchestre ces conteneurs à travers plusieurs VM.

Synthèse section 2.2

Une VM virtualise du matériel et fait tourner un OS complet. Un conteneur isole des processus sur un OS partagé. La VM offre plus d'isolation, le conteneur offre plus de vitesse et de densité. Les deux cohabitent dans les infrastructures modernes. Kubernetes, que nous verrons ensuite, orchestre des conteneurs — pas des VM.

2.3 Concepts fondamentaux de Docker

Pour travailler efficacement avec Docker, il faut maîtriser un petit nombre de concepts. Une fois ces briques assimilées, le reste n'est qu'application. Nous allons les voir un à un, en expliquant ce que c'est, à quoi ça sert et comment ça s'articule.

Docker Engine : le moteur

Docker Engine est le composant central. C'est un service (un démon) qui tourne en arrière-plan sur la machine hôte et qui répond aux requêtes que vous lui envoyez via la commande docker. Quand vous tapez docker run nginx, ce n'est pas la commande docker qui fait le travail : elle envoie une requête au démon dockerd, qui se charge ensuite de télécharger l'image, de créer les namespaces nécessaires, de configurer le réseau, et de démarrer le processus.

L'architecture est donc en client-serveur. C'est important à comprendre pour plus tard : quand on fera du Docker à distance, ou quand on se connectera au démon depuis un outil CI/CD, on exploitera cette séparation. Le client peut même être sur une autre machine que le démon.

Figure 3 — Le client docker (CLI) envoie des commandes au démon dockerd via une API REST. Le démon peut aussi communiquer avec un registry distant pour télécharger ou publier des images.

Image : le plan de construction

Une image Docker est un fichier figé qui contient tout ce qu'il faut pour exécuter une application : le système de fichiers minimal, les bibliothèques nécessaires, le code de l'application, les variables d'environnement par défaut, la commande à lancer. Une image est immuable : une fois construite, on ne la modifie plus.

Analogie utile : l'image est comme la classe en programmation orientée objet, et le conteneur est l'instance. À partir d'une même image, on peut lancer autant de conteneurs identiques qu'on veut. Chacun aura sa propre mémoire, ses propres logs, son propre état éphémère, mais tous partiront du même point de départ.

Conteneur : l'exécution vivante

Un conteneur est une instance en cours d'exécution d'une image. Il dispose de son propre système de fichiers (en superposition par-dessus celui de l'image), de ses propres processus, de sa propre interface réseau virtuelle. Il peut être démarré, arrêté, mis en pause, redémarré, supprimé. Par défaut, tout ce qui est écrit dans le conteneur est perdu quand on le supprime — sauf si on utilise un volume, que nous verrons plus loin.

Registry : le dépôt des images

Une registry est un serveur qui stocke et distribue des images Docker. La plus connue est Docker Hub (hub.docker.com), publique et gratuite pour les images open source. Il existe aussi des registries privées : Harbor, GitLab Container Registry, AWS ECR, Azure ACR, etc. Dans un contexte d'entreprise, on héberge souvent sa propre registry pour garder la main sur les images sensibles et leur disponibilité.

Le mécanisme est simple : docker pull télécharge une image depuis une registry vers votre hôte, docker push envoie une image locale vers une registry.

Dockerfile : la recette

Un Dockerfile est un fichier texte qui décrit, instruction par instruction, comment construire une image. C'est la recette reproductible par excellence : toute personne disposant du Dockerfile et du code source peut reconstruire exactement la même image. Chaque instruction du Dockerfile produit une couche (layer) dans l'image finale.

Exemple simplifié commenté :

FROM python:3.11-slim              # image de base minimaliste
WORKDIR /app                        # répertoire de travail dans l'image
COPY requirements.txt .              # copie du fichier de dépendances
RUN pip install -r requirements.txt # installation des dépendances
COPY . .                             # copie du code source
EXPOSE 5000                          # port exposé par le conteneur
CMD ["python", "app.py"]            # commande lancée au démarrage

Layers : la superposition intelligente

Une image Docker n'est pas un gros bloc monolithique. Elle est composée de couches empilées, chacune correspondant à une étape du Dockerfile. Cette architecture a deux avantages majeurs.

D'abord, les couches sont partagées entre images. Si trois images partent toutes de python:3.11-slim, la couche de base n'est téléchargée qu'une seule fois sur la machine. Économie de disque et de bande passante.

Ensuite, le cache de build tire parti des couches. Quand vous reconstruisez une image après avoir modifié uniquement votre code, Docker réutilise les couches précédentes (téléchargement, installation des dépendances) et ne refait que les étapes affectées. Gain de temps considérable en CI/CD.

Ordre des instructions dans le Dockerfile

Placez les instructions qui changent rarement en premier (installation des dépendances système, des paquets), et celles qui changent souvent à la fin (copie du code source). C'est une des bonnes pratiques les plus utiles : elle maximise l'efficacité du cache et accélère drastiquement les reconstructions.

Volume : la persistance

Par défaut, un conteneur est éphémère : ce qu'il écrit dans son système de fichiers disparaît quand on le supprime. Pour conserver des données (une base de données, des fichiers uploadés, des logs), on utilise un volume. Un volume est une zone de stockage gérée par Docker sur l'hôte, montée à l'intérieur du conteneur.

On distingue deux formes :

  • Volumes nommés : gérés par Docker, stockés dans /var/lib/docker/volumes/. À privilégier en général.
  • Bind mounts : un répertoire de l'hôte monté directement dans le conteneur. Pratique en développement, plus risqué en production.

Réseau Docker : la connectivité

Chaque conteneur dispose d'une interface réseau virtuelle. Docker crée par défaut plusieurs réseaux, dont le plus connu est bridge : les conteneurs attachés à ce réseau peuvent communiquer entre eux via leurs adresses IP internes. On peut aussi créer des réseaux personnalisés, ce qui permet notamment de faire dialoguer plusieurs conteneurs par leur nom de service plutôt que par IP.

Les types principaux :

  • bridge (par défaut) : réseau virtuel local à l'hôte. Idéal pour faire communiquer des conteneurs sur la même machine.
  • host : le conteneur partage directement la pile réseau de l'hôte. Performances maximales, isolation réseau nulle.
  • none : aucune connectivité réseau. Pour des tâches de calcul pur isolées.
  • overlay : réseau distribué entre plusieurs hôtes. Utilisé par Docker Swarm et, d'une certaine manière, par Kubernetes.

Port mapping : l'ouverture vers l'extérieur

Un conteneur qui écoute sur le port 5000 à l'intérieur n'est pas accessible de l'extérieur tant qu'on n'a pas publié ce port. Le port mapping associe un port de l'hôte à un port du conteneur. Syntaxe : -p 8080:5000 signifie « le port 8080 de l'hôte redirige vers le port 5000 du conteneur ».

C'est ce mécanisme qui permet, par exemple, d'ouvrir un navigateur sur http://ip-de-la-vm:8080 et de tomber sur une application qui, dans le conteneur, croit écouter sur 5000.

2.4 Cycle de vie d'un conteneur

Un conteneur Docker passe par plusieurs états bien définis, du moment où on le crée jusqu'à sa suppression. Comprendre ces états et les transitions entre eux est crucial, car la majorité des commandes Docker agissent précisément sur ces transitions.

Figure 4 — Les états d'un conteneur et les commandes qui provoquent les transitions.

Création

Tout commence par une image. À partir de celle-ci, Docker crée un conteneur (docker create) : il prépare le système de fichiers, réserve les ressources, configure le réseau. À ce stade, aucun processus ne tourne encore. L'état est Created.

En pratique, on utilise rarement docker create seul. On préfère docker run, qui combine create + start en une seule commande. C'est le cas d'usage le plus courant.

Exécution

docker start démarre le conteneur. Le processus principal (celui défini par CMD ou ENTRYPOINT dans le Dockerfile) est lancé. L'état passe à Running. Le conteneur est maintenant joignable via le réseau, répond aux requêtes, écrit des logs.

Durant cette phase, vous pouvez :

  • consulter les logs avec docker logs
  • entrer dans le conteneur avec docker exec -it <conteneur> bash
  • inspecter sa configuration avec docker inspect
  • suivre sa consommation avec docker stats

Arrêt

docker stop envoie un signal SIGTERM au processus principal, lui laissant dix secondes (par défaut) pour se terminer proprement. Si le processus ne répond pas, Docker envoie SIGKILL pour forcer l'arrêt. L'état devient Exited.

Un conteneur arrêté n'est pas supprimé : il existe toujours, avec son système de fichiers, prêt à être redémarré (docker start) ou supprimé (docker rm).

Suppression

docker rm efface définitivement le conteneur. Son système de fichiers en couche est libéré. Attention : les données du conteneur lui-même disparaissent, mais les volumes montés dans ce conteneur survivent (sauf option -v). C'est précisément le rôle des volumes : garantir la persistance.

Persistance via volumes : le bon réflexe

La règle mnémotechnique à retenir : si une donnée doit survivre à la suppression du conteneur, elle doit être dans un volume. Base de données, fichiers utilisateurs, logs à conserver, configurations dynamiques : tout cela va dans un volume.

Erreur fréquente — "J'ai perdu toutes mes données"

Un étudiant installe PostgreSQL dans un conteneur, y charge des données, puis fait docker rm -f postgres. Toutes les données sont perdues. C'est normal : le conteneur était éphémère. Il fallait monter un volume sur /var/lib/postgresql/data. Règle de survie : tout ce qui est précieux doit être externalisé via un volume.

2.5 Pourquoi Docker seul ne suffit pas à grande échelle

Docker résout le problème du packaging d'une application. Mais en production réelle, vous n'avez presque jamais une seule application sur une seule machine. Vous avez plusieurs applications, dépendantes les unes des autres, réparties sur plusieurs hôtes, qui doivent rester disponibles 24/7, absorber des pics de charge et se mettre à jour sans interruption. À ce stade, Docker seul atteint ses limites.

Les problèmes concrets qui émergent à l'échelle

  • Gestion de plusieurs conteneurs interdépendants. Une application moderne est rarement monolithique : une API + une base de données + un cache Redis + un worker asynchrone. Comment s'assurer qu'ils démarrent dans le bon ordre, qu'ils redémarrent si l'un tombe, qu'ils se retrouvent après un reboot ? Docker seul ne gère pas cela nativement. Il faut orchestrer.
  • Redémarrage automatique en cas de panne. Si votre conteneur crashe à 3h du matin, qui le redémarre ? Docker propose une option --restart, utile mais limitée : elle ne fonctionne que sur la machine locale. Si c'est la machine elle-même qui tombe, tout tombe avec elle.
  • Montée en charge (scaling). Un pic de trafic arrive. Vous voulez passer de 2 à 20 instances de votre API. Avec Docker seul, il faut lancer manuellement 18 conteneurs supplémentaires, reconfigurer l'équilibrage de charge, penser aux ressources. Fastidieux et sujet aux erreurs.
  • Service discovery. Dans un environnement dynamique où les conteneurs apparaissent et disparaissent, comment l'API sait-elle sur quelle IP trouver la base de données aujourd'hui ? Les IP changent à chaque redémarrage. Il faut un mécanisme d'annuaire dynamique — que Docker seul ne fournit pas de manière robuste.
  • Réseau multi-hôte. Si votre cluster a dix serveurs, comment un conteneur sur le serveur A parle-t-il à un conteneur sur le serveur E, de manière sécurisée, avec équilibrage de charge ? Docker seul reste cantonné à un hôte ; le multi-hôte exige des couches supplémentaires.
  • Stockage partagé. Si un conteneur tombe sur l'hôte A et est redémarré sur l'hôte B, il doit retrouver ses données. Un volume local à A ne suffit pas ; il faut un stockage accessible depuis tous les hôtes (NFS, Ceph, cloud storage…).
  • Haute disponibilité. Faire en sorte qu'une application reste disponible même si un serveur entier tombe demande de la réplication, du basculement automatique, du monitoring fin. Aucun de ces mécanismes n'est natif dans Docker.
  • Mise à jour sans interruption (rolling update). Déployer la version v2 sans jamais couper le service demande de lancer la v2 progressivement, de rediriger le trafic au fur et à mesure, et de pouvoir revenir en arrière (rollback) instantanément en cas de problème. Docker seul n'offre pas cela.
  • Supervision centralisée. Quand vous avez 300 conteneurs répartis sur 20 machines, vous ne pouvez plus faire docker ps partout à la main. Il faut une vue d'ensemble : logs centralisés, métriques, alertes, dashboards.
  • Gestion des secrets. Stocker des mots de passe, des clés d'API, des certificats de manière sécurisée et les distribuer uniquement aux conteneurs qui en ont besoin : problématique complexe non couverte par Docker seul.
La transition logique

C'est précisément pour répondre à l'ensemble de ces besoins — orchestration multi-conteneurs, scaling dynamique, haute disponibilité, réseau distribué, service discovery, rolling updates, gestion des secrets, supervision — que la communauté a développé des orchestrateurs de conteneurs. Docker Swarm fut une première réponse. Kubernetes, née chez Google, s'est imposée comme le standard de facto. C'est précisément l'objet de la séance 2.

Retenez bien cette articulation : Docker packagera vos applications, Kubernetes les orchestrera. Les deux technologies sont complémentaires, pas concurrentes. On construit d'abord des images Docker propres, puis on décrit à Kubernetes comment les déployer, les répliquer, les exposer, les mettre à jour.

Synthèse de la partie théorique

Vous savez maintenant pourquoi Docker existe (packaging reproductible), comment il diffère d'une VM (noyau partagé, processus isolés), quels sont ses composants essentiels (image, conteneur, registry, Dockerfile, volume, réseau), comment un conteneur évolue dans le temps (Created → Running → Exited → Supprimé), et enfin pourquoi Docker seul ne suffit pas en production à grande échelle. Tous ces concepts seront manipulés dans le TP qui suit.

3. Commandes Docker essentielles

Avant d'entrer dans le TP, voici le répertoire de commandes que nous utiliserons. Prenez le temps de lire attentivement la colonne Explication : la majorité des bugs rencontrés en TP viennent d'une incompréhension fine de ces commandes, pas d'une faute de frappe.

Commande Rôle Exemple Explication pédagogique
docker --version Afficher la version du client Docker installé docker --version Premier réflexe après installation. Confirme que la commande est disponible dans le PATH. Ne prouve pas que le démon tourne.
docker info Afficher l'état complet du démon Docker docker info Donne la version serveur, le driver de stockage, le nombre d'images et conteneurs, l'espace disque utilisé, l'OS hôte. Si ça échoue, c'est que le démon n'est pas joignable.
docker pull Télécharger une image depuis une registry docker pull nginx:1.25 Le tag (:1.25) est très important. Sans tag explicite, Docker prend latest, qui change au fil du temps. En production, toujours fixer un tag précis.
docker images Lister les images présentes localement docker images Affiche repository, tag, ID, date, taille. Utile pour vérifier ce qui est téléchargé et pour faire le ménage.
docker ps Lister les conteneurs en cours d'exécution docker ps Ne montre que les conteneurs actifs. Si votre conteneur a crashé, il n'apparaît pas ici — il faut docker ps -a.
docker ps -a Lister tous les conteneurs (y compris arrêtés) docker ps -a Indispensable pour diagnostiquer. La colonne STATUS indique la cause : Exited (0) = arrêt propre, Exited (1) = erreur, Created = jamais démarré.
docker run Créer et démarrer un conteneur à partir d'une image docker run -d -p 8080:80 nginx La commande centrale. Options clés : -d (détaché, en arrière-plan), -p (port mapping), --name (nom lisible), -e (variable d'environnement), -v (volume).
docker exec Exécuter une commande dans un conteneur actif docker exec -it web bash Permet d'entrer dans un conteneur pour inspecter, débugger. Les options -it (interactive + tty) sont obligatoires pour une session shell. Ne fonctionne pas sur un conteneur arrêté.
docker logs Afficher les logs (stdout/stderr) d'un conteneur docker logs -f web L'outil de débogage numéro un. L'option -f (follow) suit les logs en temps réel. --tail 50 pour ne voir que les 50 dernières lignes.
docker stop Arrêter proprement un conteneur actif docker stop web Envoie SIGTERM, puis SIGKILL après 10 s. À préférer à docker kill qui tue brutalement.
docker start Redémarrer un conteneur arrêté docker start web Redémarre sans recréer. Conserve la configuration initiale (ports, volumes, variables). Pratique quand on a bien configuré un conteneur et qu'on veut juste le relancer.
docker restart Arrêter puis redémarrer un conteneur docker restart web Équivaut à stop + start. Utile pour appliquer un changement de configuration externe.
docker rm Supprimer un conteneur arrêté docker rm web Échoue sur un conteneur actif. Pour forcer, ajouter -f (fait un stop + rm). Ne supprime pas l'image.
docker rmi Supprimer une image locale docker rmi nginx:1.25 Échoue si l'image est encore utilisée par un conteneur (même arrêté). Supprimer d'abord les conteneurs concernés.
docker build Construire une image à partir d'un Dockerfile docker build -t monapp:1.0 . Le point final désigne le contexte de build (répertoire courant). -t tague l'image. Chaque étape du Dockerfile produit une couche et un cache.
docker inspect Afficher la configuration complète d'un objet (JSON) docker inspect web Retourne absolument tout : réseau, volumes, variables, processus. Outil de référence pour comprendre précisément ce que Docker a configuré.
docker volume ls Lister les volumes Docker docker volume ls Affiche les volumes nommés. Combiné avec docker volume inspect <nom>, donne le chemin physique sur l'hôte.
docker network ls Lister les réseaux Docker docker network ls Montre au minimum bridge, host, none. Permet de voir les réseaux personnalisés créés pour isoler des groupes de conteneurs.
Conseils d'apprentissage pour ce vocabulaire

Ne cherchez pas à mémoriser toutes ces commandes par cœur d'un coup. En revanche, retenez l'intention de chacune. En TP, vous utiliserez naturellement run, ps, logs, exec, stop, rm dizaines de fois : la mémoire musculaire se construira d'elle-même. Les autres commandes se retrouvent facilement avec docker --help ou docker <commande> --help.

4. L'application fil rouge du TP

Pour donner de la cohérence au TP, nous allons travailler sur une même petite application tout au long des exercices. Cette application est volontairement minimaliste : elle tient en une trentaine de lignes de Python. L'objectif n'est pas de faire du Flask avancé, mais d'avoir un support concret pour illustrer chaque concept Docker.

Présentation du service « DevOps Monitor »

Imaginons un petit service web qui servirait d'indicateur de santé basique dans une infrastructure. Trois fonctionnalités :

  • Une page d'accueil (/) qui affiche un message personnalisable et incrémente un compteur à chaque visite.
  • Une route de santé (/health) qui retourne un JSON avec l'état du service, l'heure, le hostname. C'est l'équivalent d'un health check tel qu'on l'utilise en supervision.
  • Une route de statistiques (/stats) qui expose le compteur de visites et le nombre d'écritures en fichier de log.

L'application utilise :

  • le framework Flask pour la partie web,
  • une variable d'environnement APP_MESSAGE pour personnaliser le message d'accueil (illustrera la configuration par variable),
  • un fichier de log dans un répertoire dédié pour enregistrer chaque visite (illustrera la persistance via volume).

Intérêt pédagogique

Pourquoi ce choix d'application ? Parce qu'elle permet d'illustrer, de manière ramassée mais réaliste, l'ensemble des concepts Docker que nous voulons pratiquer :

Concept DockerIllustration dans l'application
Image de baseHéritage depuis python:3.11-slim
Installation de dépendancesFlask via pip install dans le Dockerfile
Copie du codeLe fichier app.py est intégré dans l'image
Commande de démarrageDirective CMD qui lance le serveur Flask
Port exposéFlask écoute sur 5000, on mappera sur 8080 côté hôte

Pont vers Kubernetes

La route /health n'est pas anodine : en Kubernetes, chaque Pod définit généralement un liveness probe et un readiness probe, c'est-à-dire deux endpoints que Kubernetes interroge régulièrement pour savoir si le conteneur est vivant et prêt à servir du trafic. En développant l'habitude dès maintenant, vous préparez la séance 2 sans effort.

Pourquoi pas une application plus complexe ?

Parce que la complexité noierait le propos. L'objectif de ce TP n'est pas d'apprendre Flask, mais Docker. Une application volontairement simple permet de se concentrer sur ce qui change réellement d'un exercice à l'autre : un port, une variable, un volume. Vous pourrez plus tard refaire le même exercice avec vos propres applications Python, Node.js ou Java.

5. TP guidé complet

Cette partie occupe environ 2h15. Chaque exercice suit la même structure : objectif, prérequis, étapes commentées, résultats attendus, points de vigilance. Respectez l'ordre : chaque TP s'appuie sur le précédent.

Contexte du lab

L'ensemble du TP se déroule dans une VM Ubuntu Server 22.04 (ou 24.04) LTS déployée sur l'infrastructure ESXi de votre laboratoire. Configuration recommandée : 2 vCPU, 4 Go de RAM, 30 Go de disque, un accès internet. Vous y êtes connecté en SSH avec un compte disposant des droits sudo.

TP 1 — Vérification de l'environnement

⏱️ 10 min
Objectif

Vérifier que la VM est prête à recevoir Docker : OS à jour, accès internet, droits suffisants, ressources disponibles.

Étape 1.1 — Se connecter à la VM et vérifier l'OS

Depuis votre poste, connectez-vous en SSH à la VM Ubuntu. Une fois connecté, identifiez la version exacte de l'OS : cela conditionnera les dépôts Docker utilisés.

$ lsb_release -a
$ uname -r
Résultat attendu

Une distribution Ubuntu récente (22.04 ou 24.04) et un noyau Linux 5.x ou 6.x. Si la distribution est plus ancienne, certains paquets Docker peuvent manquer.

Étape 1.2 — Vérifier l'accès réseau

Docker va devoir télécharger des images depuis Docker Hub. Un simple test de connectivité suffit.

$ ping -c 3 hub.docker.com
$ curl -I https://registry-1.docker.io/v2/
Résultat attendu

Le ping doit répondre (sauf si ICMP est filtré — dans ce cas, seul le curl compte). Le curl doit renvoyer un code HTTP (401 est normal, cela signifie que le serveur répond mais demande une authentification pour aller plus loin).

Étape 1.3 — Vérifier les droits sudo
$ sudo -v
$ id

La commande sudo -v valide vos droits (demande le mot de passe si nécessaire). id affiche les groupes de votre utilisateur.

Étape 1.4 — Vérifier les ressources disponibles
$ free -h
$ df -h /
$ nproc
Résultat attendu

Au moins 2 Go de RAM libre, 10 Go d'espace disque libre, 2 CPU visibles. En-dessous, le TP reste faisable mais certains builds seront lents.

Point de vigilance

Si vous n'avez pas d'accès internet (proxy d'entreprise restrictif), configurez le proxy avant toute installation. Docker a besoin de joindre registry-1.docker.io et auth.docker.io au minimum.

TP 2 — Installation de Docker

⏱️ 15 min
Objectif

Installer Docker Engine en utilisant les dépôts officiels, configurer l'accès sans sudo pour votre utilisateur, vérifier que le service démarre correctement.

Étape 2.1 — Mettre à jour le système et installer les prérequis
$ sudo apt update
$ sudo apt install -y ca-certificates curl gnupg lsb-release

Ces paquets permettront de récupérer la clé GPG officielle de Docker et d'ajouter son dépôt APT en HTTPS de manière sécurisée.

Étape 2.2 — Ajouter la clé GPG officielle Docker
$ sudo install -m 0755 -d /etc/apt/keyrings
$ curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
      sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
$ sudo chmod a+r /etc/apt/keyrings/docker.gpg

On télécharge la clé publique de Docker, on la convertit au format binaire attendu par APT, et on la place dans un répertoire dédié. Cette clé servira à vérifier l'authenticité des paquets Docker téléchargés.

Étape 2.3 — Ajouter le dépôt Docker stable
$ echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
$ sudo apt update

Le dépôt Docker est maintenant reconnu par APT. La commande lsb_release -cs retourne le nom de code de votre distribution (jammy pour 22.04, noble pour 24.04), et Docker fournit un dépôt spécifique pour chacun.

Étape 2.4 — Installer Docker Engine et le plugin Compose
$ sudo apt install -y docker-ce docker-ce-cli containerd.io \
      docker-buildx-plugin docker-compose-plugin

Cinq paquets sont installés :

  • docker-ce : le démon Docker (Community Edition)
  • docker-ce-cli : le client en ligne de commande
  • containerd.io : le runtime de conteneur bas niveau utilisé par Docker
  • docker-buildx-plugin : builder multi-architecture
  • docker-compose-plugin : la commande docker compose (v2)
Étape 2.5 — Vérifier que le service est actif
$ sudo systemctl status docker
$ sudo systemctl enable docker

La première commande affiche l'état du service (doit être active (running)). La seconde active le démarrage automatique au boot.

Étape 2.6 — Utiliser Docker sans sudo (optionnel mais recommandé)
$ sudo usermod -aG docker $USER
$ newgrp docker
$ docker version

On ajoute l'utilisateur courant au groupe docker. La commande newgrp applique immédiatement ce changement de groupe dans la session courante. Ensuite, docker version devrait fonctionner sans sudo.

Sécurité : conséquence d'ajouter un utilisateur au groupe docker

Le groupe docker confère à ses membres des droits équivalents à root : on peut monter / dans un conteneur et écrire n'importe où. Sur un poste personnel ou une VM de TP, c'est acceptable. En production ou sur un serveur partagé, réfléchissez-y à deux fois et utilisez plutôt sudo docker.

Étape 2.7 — Vérification finale
$ docker --version
$ docker info | head -20
Résultat attendu

Une version Docker affichée (ex. Docker version 27.x.x) et une sortie de docker info qui mentionne Server Version avec un numéro, Storage Driver: overlay2, et l'OS hôte. Aucune erreur de connexion au démon.

Erreur fréquente : « Cannot connect to the Docker daemon »

Si cette erreur apparaît, deux causes principales. Soit le service n'est pas démarré (sudo systemctl start docker), soit votre utilisateur n'est pas dans le groupe docker et n'a pas relancé sa session. La commande newgrp docker résout le second cas dans le shell courant ; pour un effet permanent, déconnectez-vous puis reconnectez-vous.

TP 3 — Premier conteneur : hello-world

⏱️ 10 min
Objectif

Exécuter son tout premier conteneur et comprendre ce qui se passe en arrière-plan : téléchargement de l'image, création du conteneur, exécution, message, terminaison.

Étape 3.1 — Lancer hello-world
$ docker run hello-world

Observez attentivement la sortie. Elle explique exactement les étapes que Docker vient de réaliser :

Unable to find image 'hello-world:latest' locally
latest: Pulling from library/hello-world
...
Status: Downloaded newer image for hello-world:latest

Hello from Docker!
This message shows that your installation appears to be working correctly.

To generate this message, Docker took the following steps:
 1. The Docker client contacted the Docker daemon.
 2. The daemon pulled the "hello-world" image from the Docker Hub.
 3. The daemon created a new container from that image which runs the
    executable that produces the output you are currently reading.
 4. The daemon streamed that output to the Docker client, which sent it
    to your terminal.
Que s'est-il passé exactement ?

Cette sortie est pédagogique à dessein. Elle décrit précisément les quatre étapes :

  1. Le client (votre docker run) a contacté le démon dockerd.
  2. Le démon n'avait pas l'image localement, il l'a téléchargée depuis Docker Hub.
  3. Il a créé un conteneur à partir de cette image.
  4. Il a exécuté le programme embarqué dans l'image, dont la sortie est redirigée vers votre terminal.

Vous venez d'utiliser, sans le savoir, toute la chaîne décrite dans la figure 3 de la partie théorique.

Étape 3.2 — Vérifier l'image téléchargée
$ docker images

L'image hello-world apparaît. Notez sa taille : quelques kilooctets seulement. C'est l'image Docker la plus petite de l'écosystème, pensée précisément pour être un test.

Étape 3.3 — Inspecter le conteneur
$ docker ps
$ docker ps -a

La première commande ne renvoie rien : le conteneur hello-world a terminé son exécution et n'est plus actif. La seconde le montre avec un statut Exited (0) : arrêté normalement avec un code de retour 0 (succès).

Conteneur = cycle de vie court par nature

Un conteneur ne tourne que tant que son processus principal tourne. Pour hello-world, ce processus affiche un message puis se termine. Le conteneur s'arrête donc immédiatement. Pour un serveur web, le processus reste actif en permanence, et le conteneur reste dans l'état Running. C'est la clé du comportement Docker.

Étape 3.4 — Nettoyage
$ docker ps -a
# noter le CONTAINER ID ou le NAME du conteneur hello-world
$ docker rm <container_id>

Le conteneur est supprimé. L'image, elle, reste (nous la laisserons pour l'instant).

TP 4 — Manipulation d'images et de conteneurs

⏱️ 15 min
Objectif

Prendre en main les commandes quotidiennes : télécharger une image explicitement, lancer un conteneur en mode interactif, entrer dedans, lire ses logs, l'arrêter proprement, le supprimer.

Étape 4.1 — Télécharger explicitement une image
$ docker pull alpine:3.20

Alpine Linux est une distribution minimaliste (5 Mo environ). On spécifie le tag 3.20 pour une version précise, ce qui est une bonne pratique.

$ docker images

L'image alpine apparaît avec sa taille. Retenez : toujours fixer un tag précis plutôt que de laisser latest par défaut.

Étape 4.2 — Lancer un conteneur interactif
$ docker run -it --name alpine-test alpine:3.20 sh

Décortiquons cette commande :

  • -i : mode interactif (garde stdin ouvert)
  • -t : alloue un pseudo-terminal
  • --name alpine-test : donne un nom au conteneur (plus lisible que l'ID aléatoire)
  • alpine:3.20 : l'image à utiliser
  • sh : la commande à exécuter dans le conteneur (un shell)

Vous arrivez dans un shell à l'intérieur du conteneur. Le prompt change, par exemple / #.

Étape 4.3 — Explorer l'intérieur du conteneur
/ # cat /etc/os-release
/ # ls /
/ # ps aux
/ # hostname
Observation importante

ps aux ne montre que deux processus : le shell lui-même et la commande ps. C'est la démonstration concrète des namespaces PID : le conteneur ne voit pas les processus de l'hôte. Il se croit seul. L'isolation est réelle.

Le hostname est un identifiant court (souvent les 12 premiers caractères de l'ID du conteneur). Là encore, namespace UTS : le conteneur a son propre nom.

Sortez du conteneur :

/ # exit

Le shell se termine, donc le processus principal se termine, donc le conteneur s'arrête. Vérifiez :

$ docker ps -a | grep alpine-test
Étape 4.4 — Lancer un conteneur en arrière-plan
$ docker run -d --name ping-box alpine:3.20 sh -c "while true; do echo 'je suis vivant'; sleep 2; done"

Explications :

  • -d : mode détaché, le conteneur tourne en arrière-plan
  • la commande sh -c "..." exécute une boucle infinie qui écrit un message toutes les 2 secondes

Le conteneur tourne maintenant en arrière-plan. Vérifiez :

$ docker ps
Étape 4.5 — Lire les logs
$ docker logs ping-box
$ docker logs --tail 5 ping-box
$ docker logs -f ping-box     # Ctrl+C pour arrêter le suivi

Les logs d'un conteneur sont simplement ce que son processus principal écrit sur stdout et stderr. Pas de fichier spécial, pas de magie. C'est pour cela qu'il est recommandé que les applications conteneurisées écrivent leurs logs sur la sortie standard plutôt que dans des fichiers.

Étape 4.6 — Exécuter une commande dans un conteneur actif
$ docker exec -it ping-box sh

Vous entrez dans le conteneur pendant qu'il tourne. C'est l'outil de débogage par excellence.

/ # ps aux
/ # exit

Observez que vous voyez maintenant la boucle while true et votre propre shell. En sortant (exit), vous fermez votre session exec — mais le conteneur continue de tourner, car son processus principal (la boucle) est toujours actif.

Étape 4.7 — Arrêter et supprimer
$ docker stop ping-box
$ docker ps -a
$ docker rm ping-box
$ docker rm alpine-test

stop envoie SIGTERM puis SIGKILL après 10 s. rm supprime le conteneur. L'image alpine reste en place pour la suite du TP.

Erreur fréquente — Oublier le -it

Beaucoup d'étudiants tapent docker run alpine sh (sans -it) et sont surpris que le conteneur se termine instantanément. C'est normal : sans TTY, sh n'a pas d'entrée interactive, il lit EOF et se termine. Retenez : pour un shell interactif, -it est obligatoire. Pour un démon en arrière-plan, c'est -d et surtout pas -it.

TP 5 — Exécution d'un serveur web simple

⏱️ 10 min
Objectif

Lancer un serveur web (nginx) dans un conteneur, publier son port vers l'hôte, tester l'accès depuis l'extérieur. Comprendre en profondeur le mécanisme de port mapping.

Étape 5.1 — Lancer nginx en arrière-plan
$ docker run -d --name web -p 8080:80 nginx:1.27-alpine

Décortiquons l'option -p 8080:80 : elle signifie « tout le trafic arrivant sur le port 8080 de la VM sera redirigé vers le port 80 du conteneur ». Le conteneur, lui, croit écouter sur 80. De l'extérieur, on attaque la VM sur 8080.

Étape 5.2 — Tester depuis la VM elle-même
$ curl http://localhost:8080
$ curl -I http://localhost:8080

La première commande affiche le HTML de la page d'accueil nginx. La seconde ne montre que les en-têtes HTTP, y compris Server: nginx/1.27.x.

Étape 5.3 — Tester depuis un navigateur extérieur

Récupérez l'adresse IP de votre VM :

$ ip -4 addr show | grep inet

Depuis un navigateur sur votre poste, ouvrez http://<ip-de-la-vm>:8080. La page de bienvenue nginx doit s'afficher.

Si ça ne marche pas depuis l'extérieur

Vérifiez d'abord que le pare-feu de la VM ne bloque pas le port 8080. Sur Ubuntu avec UFW : sudo ufw status. Vérifiez ensuite les règles du réseau ESXi (VLAN, port group). Enfin, assurez-vous que votre poste et la VM sont bien sur le même réseau logique.

Étape 5.4 — Observer les logs d'accès

Dans un terminal, laissez tourner :

$ docker logs -f web

Puis depuis un autre terminal (ou votre navigateur), rechargez la page plusieurs fois. Chaque requête apparaît en temps réel dans les logs. C'est exactement ce que fait un observateur de logs en production — sur un cluster entier, ces flux seront centralisés par un outil comme Loki ou ELK.

Étape 5.5 — Inspecter la configuration réseau du conteneur
$ docker inspect web --format='{{json .NetworkSettings}}' | python3 -m json.tool

Vous voyez l'IP interne du conteneur (par exemple 172.17.0.2), sa MAC, la gateway, le mapping de port. Cela confirme qu'il y a bien un réseau virtuel entre l'hôte et le conteneur.

Étape 5.6 — Nettoyage
$ docker stop web
$ docker rm web
Synthèse intermédiaire des TP 1 à 5

TP 6 — Création de l'application fil rouge

⏱️ 15 min
Objectif

Créer localement l'application Flask qui nous accompagnera dans tous les TP suivants. Comprendre sa structure avant de la conteneuriser.

Étape 6.1 — Créer l'arborescence du projet
$ mkdir -p ~/devops-monitor

Étape 6.3 — Créer le fichier app.py

Créez le fichier avec votre éditeur préféré (nano app.py) et collez le contenu suivant :

from flask import Flask, jsonify
import os, socket, datetime, threading

app = Flask(__name__)
lock = threading.Lock()

MESSAGE = os.environ.get("APP_MESSAGE", "Bonjour depuis DevOps Monitor !")
LOG_DIR = "/app/logs"
LOG_FILE = os.path.join(LOG_DIR, "visits.log")

visits = 0
os.makedirs(LOG_DIR, exist_ok=True)

def record_visit(path):
    global visits
    with lock:
        visits += 1
        with open(LOG_FILE, "a") as f:
            f.write(f"{datetime.datetime.utcnow().isoformat()} {path}\n")

@app.route("/")
def home():
    record_visit("/")
    return f"""
    <html><body style='font-family:sans-serif;padding:40px'>
    <h1>{MESSAGE}</h1>
    <p>Conteneur : <b>{socket.gethostname()}</b></p>
    <p>Visites : <b>{visits}</b></p>
    <p><a href='/health'>/health</a> · <a href='/stats'>/stats</a></p>
    </body></html>
    """

@app.route("/health")
def health():
    return jsonify({
        "status": "ok",
        "hostname": socket.gethostname(),
        "utc": datetime.datetime.utcnow().isoformat() + "Z"
    })

@app.route("/stats")
def stats():
    log_lines = 0
    if os.path.exists(LOG_FILE):
        with open(LOG_FILE) as f:
            log_lines = sum(1 for _ in f)
    return jsonify({"visits": visits, "log_lines": log_lines})

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
Étape 6.4 — Comprendre chaque bloc

Lisons l'application bloc par bloc :

  • Imports et setup. Flask pour le web, os pour les variables d'environnement, socket pour le hostname du conteneur, datetime pour l'horodatage.
  • Variable MESSAGE. Récupérée depuis la variable d'environnement APP_MESSAGE. Si celle-ci n'est pas définie, une valeur par défaut est utilisée. Ce point est crucial — il illustre comment une image Docker peut être configurée différemment selon son contexte d'exécution, sans modifier l'image elle-même.
  • Variable LOG_DIR. Pointée vers /app/logs. C'est le répertoire à l'intérieur du conteneur. Nous y écrirons un fichier à chaque visite. Plus tard, on montera un volume dessus pour garantir la persistance.
  • Routes. / affiche une page HTML, /health renvoie un JSON de statut (typique des health checks Kubernetes), /stats expose des métriques.
  • host="0.0.0.0". Obligatoire en conteneur : sans cela, Flask n'écouterait que sur 127.0.0.1 à l'intérieur du conteneur, inaccessible depuis l'extérieur.
Le piège classique : host="127.0.0.1"

Un des débuggages les plus frustrants pour un débutant : « mon appli marche sur mon poste, pas dans le conteneur ». La cause est presque toujours la même : l'application écoute sur 127.0.0.1, c'est-à-dire la boucle locale du conteneur. Depuis l'hôte, on ne peut pas l'atteindre. Il faut obligatoirement host="0.0.0.0" dans Flask, ou l'équivalent dans d'autres frameworks.

TP 7 — Écriture du Dockerfile

⏱️ 15 min
Objectif

Écrire un Dockerfile propre, comprendre chaque instruction, appliquer les bonnes pratiques de base (ordre des instructions, image slim, utilisateur non-root).

Étape 7.1 — Créer le Dockerfile

Depuis ~/devops-monitor, créez le fichier Dockerfile (sans extension) :

# ---------- Image de base ----------
FROM python:3.11-slim

# ---------- Métadonnées ----------
LABEL maintainer="cours-cloud@ensa.ma"
LABEL description="Application fil rouge - séance 1 Docker"

# ---------- Répertoire de travail ----------
WORKDIR /app

# ---------- Dépendances (étape stable en premier) ----------
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# ---------- Code applicatif ----------
COPY app.py .

# ---------- Répertoire de logs (persistable) ----------
RUN mkdir -p /app/logs

# ---------- Utilisateur non-root ----------
RUN useradd --create-home --shell /bin/bash appuser \
    && chown -R appuser:appuser /app
USER appuser

# ---------- Port exposé et commande ----------
EXPOSE 5000
CMD ["python", "app.py"]
Étape 7.2 — Explication ligne par ligne
InstructionExplication pédagogique
FROM python:3.11-slimHérite d'une image officielle Python minimaliste (~130 Mo). La variante slim ne contient que l'essentiel, sans les outils de développement. Pour un conteneur encore plus léger, on pourrait utiliser python:3.11-alpine.
LABEL ...Ajoute des métadonnées à l'image. Utile pour la traçabilité, notamment en environnement d'entreprise. Purement informatif, sans effet sur l'exécution.
WORKDIR /appDéfinit le répertoire de travail. Toutes les instructions suivantes (COPY, RUN) s'exécutent dans ce répertoire. Crée le répertoire s'il n'existe pas.
COPY requirements.txt .Copie uniquement le fichier de dépendances (pas encore le code). Cette séparation est volontaire : elle permet au cache Docker de réutiliser l'installation des dépendances tant que requirements.txt ne change pas, même si le code change.
RUN pip install --no-cache-dir ...Installe les dépendances. L'option --no-cache-dir évite que pip conserve ses caches dans l'image finale, économisant quelques Mo.
COPY app.py .Copie le code après les dépendances. Si on change le code mais pas les dépendances, seule cette dernière couche sera reconstruite : gain de temps énorme en CI/CD.
RUN mkdir -p /app/logsCrée le répertoire de logs qui pourra être monté en volume plus tard.
RUN useradd ... + USER appuserCrée un utilisateur non-root et bascule l'exécution sous ce compte. Par défaut, un conteneur tourne en root, ce qui est un risque de sécurité. Bonne pratique en production.
EXPOSE 5000Documente le fait que le conteneur écoute sur le port 5000. N'ouvre pas réellement le port — c'est -p à l'exécution qui fait cela. C'est une indication, lue par les orchestrateurs et les outils.
CMD ["python", "app.py"]Commande exécutée au démarrage du conteneur. La forme JSON (tableau) est préférée à la forme shell : elle ne passe pas par un shell intermédiaire, donc les signaux (SIGTERM) atteignent bien le processus Python.
Étape 7.3 — Pourquoi cet ordre précis ?

L'ordre des instructions dans un Dockerfile n'est pas cosmétique : il détermine l'efficacité du cache de build. Retenez le principe suivant :

  • Les instructions qui changent rarement (image de base, installation système) en haut.
  • Les instructions intermédiaires (dépendances applicatives) au milieu.
  • Les instructions qui changent souvent (copie du code source) en bas.

Ainsi, quand un développeur change une ligne de code, Docker ne refait que la dernière couche. Construction en quelques secondes au lieu de quelques minutes.

Bonnes pratiques du Dockerfile

Retenez ces six règles d'or : (1) fixer des versions précises, pas latest ; (2) utiliser une image de base slim ou alpine ; (3) placer les dépendances avant le code ; (4) tourner en utilisateur non-root ; (5) une seule responsabilité par conteneur ; (6) documenter avec EXPOSE et LABEL.

Étape 7.4 — Créer un fichier .dockerignore

Pour éviter d'envoyer des fichiers inutiles au démon lors du build, créez un .dockerignore :

logs/
__pycache__/
*.pyc
.git/
.vscode/
.DS_Store

Ces fichiers ne seront pas copiés dans l'image, même s'ils sont présents dans le répertoire du projet. L'image sera plus propre et plus légère.

Étape 8.4 — Tester le cache de build

Refaites un build sans rien changer :

$ docker build -t devops-monitor:1.0 .

Observez : chaque étape affiche CACHED. Le build prend moins d'une seconde. C'est la puissance du cache Docker.

TP 10 — Injecter des variables d'environnement

⏱ 10 min

Dans un monde conteneurisé, on ne modifie pas le code pour changer le comportement d'une application : on modifie des variables d'environnement. C'est exactement le principe du 12-Factor App qui sera à la base des ConfigMap et Secret que vous rencontrerez en Kubernetes. Dans cette étape, nous allons voir à quel point il est simple de reconfigurer notre application fil rouge sans rebuilder l'image.

Étape 1 — Arrêter et supprimer le conteneur précédent

Nous avions laissé tourner le conteneur monitor à la fin du TP 9. Comme nous voulons relancer la même image avec une configuration différente, il faut d'abord libérer le nom. 

$ docker stop monitor
$ docker rm monitor
Étape 2 — Relancer avec une variable d'environnement personnalisée

L'option -e (ou --env) permet d'injecter une variable au moment du docker run. L'application Flask la lira via os.environ.get("APP_MESSAGE", ...), comme prévu dans le code du TP 6. 

$ docker run -d \
    --name monitor \
    -p 8080:5000 \
    -e APP_MESSAGE="Bienvenue à l'ENSA Fès - Promo Cloud 2026" \
    devops-monitor:1.0
Résultat attendu. Le conteneur démarre, et la commande curl http://localhost:8080/ renvoie désormais la nouvelle phrase. Aucun rebuild, aucun changement de code : même image, comportement différent.
Étape 3 — Vérifier la valeur injectée
$ curl http://localhost:8080/
$ curl http://localhost:8080/health

Vous pouvez aussi vérifier directement les variables d'environnement que Docker a injectées à l'intérieur du conteneur : 

$ docker exec monitor env | grep APP_MESSAGE
Étape 4 — Lancer une seconde instance avec une autre configuration

C'est ici que l'intérêt devient concret : la même image peut produire autant de conteneurs différents que nécessaire, chacun avec sa propre configuration. Cela préfigure les replicas Kubernetes que vous verrez bientôt. 

$ docker run -d \
    --name monitor-prod \
    -p 8081:5000 \
    -e APP_MESSAGE="Instance PROD" \
    devops-monitor:1.0

$ docker run -d \
    --name monitor-dev \
    -p 8082:5000 \
    -e APP_MESSAGE="Instance DEV" \
    devops-monitor:1.0
$ curl http://localhost:8081/     # renvoie "Instance PROD"
$ curl http://localhost:8082/     # renvoie "Instance DEV"
Bonne pratique. Ne jamais coder en dur une URL de base de données, un mot de passe ou une clé d'API dans un Dockerfile. On passe toujours ces valeurs via -e, un fichier --env-file, ou — en Kubernetes — un ConfigMap / Secret.
Erreur fréquente. Beaucoup d'étudiants essaient de changer la valeur avec docker exec monitor export APP_MESSAGE=.... Cela ne fonctionne pas : les variables d'environnement sont fixées au démarrage du conteneur. Pour les changer, il faut arrêter puis relancer le conteneur.
À retenir. Une image est un template immuable. La configuration vit à l'extérieur de l'image, injectée au runtime. Ce découplage est la clé de la portabilité : la même image tourne en dev, recette et production, simplement en changeant les variables.

Avant de passer au TP 11, nettoyons les deux instances supplémentaires :

$ docker stop monitor-prod monitor-dev
$ docker rm monitor-prod monitor-dev

TP 11 — Volumes et persistance des données

⏱ 15 min

Souvenez-vous du constat de la fin du TP 9 : lorsque le conteneur est supprimé, le fichier /app/logs/visits.log disparaît avec lui. C'est normal : un conteneur est éphémère, son système de fichiers est construit sur des couches en lecture seule plus une couche écrivable qui vit et meurt avec le conteneur. Pour conserver les données, il faut les externaliser dans un volume.

Mise en perspective. En Kubernetes, vous retrouverez exactement ce concept sous le nom de PersistentVolume et PersistentVolumeClaim. La logique que vous apprenez ici est directement transférable.
Étape 1 — Démonstration du problème (sans volume)

Commençons par reproduire clairement le problème, pour bien comprendre ce que résout le volume. 

$ docker stop monitor && docker rm monitor

$ docker run -d --name monitor -p 8080:5000 devops-monitor:1.0

# On génère quelques visites
$ curl http://localhost:8080/
$ curl http://localhost:8080/
$ curl http://localhost:8080/

# On vérifie que le log existe dans le conteneur
$ docker exec monitor cat /app/logs/visits.log

# On détruit le conteneur
$ docker stop monitor && docker rm monitor

# On en recrée un nouveau à partir de la même image
$ docker run -d --name monitor -p 8080:5000 devops-monitor:1.0

# Et... le log est vide !
$ docker exec monitor cat /app/logs/visits.log
# → cat: /app/logs/visits.log: No such file or directory
Observation pédagogique. Les données écrites à l'intérieur du conteneur sont perdues dès sa suppression. Ce comportement n'est pas un bug : c'est une garantie de reproductibilité. Mais il faut un mécanisme pour conserver ce qui doit survivre.
Étape 2 — Créer un volume Docker nommé

Un volume nommé est un espace de stockage géré par Docker, stocké sur l'hôte dans /var/lib/docker/volumes/, et monté dans le conteneur à l'endroit qu'on choisit. 

$ docker stop monitor && docker rm monitor

$ docker volume create monitor-logs

$ docker volume ls
# DRIVER    VOLUME NAME
# local     monitor-logs

$ docker volume inspect monitor-logs

Le volume existe, mais il est vide et pas encore attaché à quoi que ce soit.

Étape 3 — Relancer l'application en montant le volume

L'option -v <nom_volume>:<chemin_dans_conteneur> attache le volume au conteneur. Tout ce qui sera écrit dans /app/logs sera en réalité écrit dans le volume monitor-logs. 

$ docker run -d \
    --name monitor \
    -p 8080:5000 \
    -v monitor-logs:/app/logs \
    devops-monitor:1.0
# On génère quelques visites
$ curl http://localhost:8080/
$ curl http://localhost:8080/
$ curl http://localhost:8080/
$ curl http://localhost:8080/

# On vérifie le log
$ docker exec monitor cat /app/logs/visits.log
Étape 4 — Test de persistance : détruire et recréer

C'est le moment de vérité. On détruit le conteneur, on en recrée un nouveau attaché au même volume, et on observe. 

$ docker stop monitor && docker rm monitor

$ docker run -d \
    --name monitor \
    -p 8080:5000 \
    -v monitor-logs:/app/logs \
    devops-monitor:1.0

# Les données sont toujours là !
$ docker exec monitor cat /app/logs/visits.log
Résultat attendu. Les lignes de log générées avant la destruction du premier conteneur sont toujours présentes dans le nouveau. Le volume a survécu à la disparition du conteneur : il a fait exactement son travail.
Étape 5 — Inspecter le volume depuis l'hôte

Pour démystifier le mécanisme, allons voir où Docker stocke réellement les données sur la VM hôte. 

$ sudo ls -la /var/lib/docker/volumes/monitor-logs/_data/
$ sudo cat /var/lib/docker/volumes/monitor-logs/_data/visits.log
Étape 6 — Comparaison : volume nommé vs bind mount

Il existe une seconde forme de montage, le bind mount, qui monte un répertoire arbitraire de l'hôte dans le conteneur. Pratique en développement, mais moins portable. 

$ mkdir -p ~/monitor-data

$ docker run -d \
    --name monitor-bind \
    -p 8090:5000 \
    -v ~/monitor-data:/app/logs \
    devops-monitor:1.0

$ curl http://localhost:8090/
$ cat ~/monitor-data/visits.log

Mini-synthèse. Un conteneur est éphémère, un volume est persistant. On choisit explicitement ce qu'on veut faire survivre aux destructions de conteneurs. Cette séparation entre exécution et données est l'une des idées les plus importantes de la conteneurisation.

Avant de continuer, on nettoie le conteneur de démonstration bind :

$ docker stop monitor-bind && docker rm monitor-bind

TP 12 — Réseau Docker et communication entre conteneurs

⏱ 15 min

Jusqu'ici, nous avons fait dialoguer l'hôte avec un conteneur via le port mapping. Mais dans une application réelle, un backend doit parler à une base de données, un frontend à une API, etc. Docker fournit un mécanisme simple : les réseaux Docker.

Figure 5 — Deux conteneurs attachés à un réseau bridge personnalisé peuvent se joindre par leur nom de conteneur, grâce au DNS interne fourni par Docker.
Étape 1 — Lister les réseaux existants
$ docker network ls

Vous voyez au moins trois réseaux par défaut :

  • bridge : le réseau par défaut des conteneurs. Suffisant mais sans DNS par nom de conteneur.
  • host : le conteneur partage directement la pile réseau de l'hôte.
  • none : aucun réseau, isolation totale.
Étape 2 — Créer un réseau bridge personnalisé

Créer son propre réseau bridge présente un avantage majeur par rapport au bridge par défaut : Docker y active automatiquement la résolution DNS par nom de conteneur. Les conteneurs peuvent se joindre par leur nom, exactement comme des services Kubernetes. 

$ docker network create app-network

$ docker network inspect app-network
Étape 3 — Lancer l'API sur ce réseau
$ docker stop monitor 2>/dev/null ; docker rm monitor 2>/dev/null

$ docker run -d \
    --name api \
    --network app-network \
    -v monitor-logs:/app/logs \
    -e APP_MESSAGE="API derrière réseau personnalisé" \
    devops-monitor:1.0
Notez bien. Nous n'avons pas publié de port avec -p. Dans ce scénario, l'API n'est pas accessible depuis l'hôte — uniquement depuis les autres conteneurs du même réseau. C'est volontaire : cela illustre le principe d'un service interne.
Étape 4 — Lancer un conteneur client sur le même réseau

On va créer un second conteneur, basé sur Alpine, qui va joindre l'API par son nom. 

$ docker run -it --rm \
    --name web \
    --network app-network \
    alpine:3.20 sh

À l'intérieur du conteneur web, on installe curl et on teste :

/ # apk add --no-cache curl
/ # curl http://api:5000/
/ # curl http://api:5000/health
/ # curl http://api:5000/stats
/ # exit
Résultat attendu. Les trois requêtes répondent correctement. Remarquez que nous avons utilisé api comme nom d'hôte — pas d'IP, pas de localhost. Docker a fait la résolution DNS pour nous.
Étape 5 — Démonstration de l'isolation

Prouvons que les réseaux Docker isolent réellement. On crée un second conteneur, sans le joindre à app-network : il ne pourra pas joindre api. 

$ docker run -it --rm alpine:3.20 sh
/ # apk add --no-cache curl
/ # curl --max-time 3 http://api:5000/
# → curl: (6) Could not resolve host: api
/ # exit
Pont vers Kubernetes. Ce que vous venez de faire — isoler des conteneurs par réseau, leur donner des noms DNS stables, n'exposer qu'une partie à l'extérieur — c'est exactement ce que fait Kubernetes avec ses Service, ses NetworkPolicy et ses namespaces.
Mini-synthèse. Les réseaux Docker fournissent trois choses essentielles : l'isolation, la résolution DNS par nom, et la possibilité de décider qui communique avec qui. Ce sont les fondations conceptuelles du réseau Kubernetes.

TP 13 — Nettoyage de l'environnement

⏱ 10 min

Un bon administrateur laisse toujours son environnement propre. Docker a tendance à accumuler rapidement des conteneurs arrêtés, des images intermédiaires et des volumes orphelins qui consomment plusieurs Go sans qu'on s'en rende compte. Voyons comment remettre de l'ordre proprement.

Étape 1 — État des lieux
$ docker ps -a
$ docker images
$ docker volume ls
$ docker network ls
$ docker system df

La commande docker system df est particulièrement utile : elle vous indique combien d'espace est occupé par chaque type d'objet et combien est récupérable.

Étape 2 — Arrêter et supprimer tous les conteneurs du TP
$ docker stop api 2>/dev/null
$ docker rm api 2>/dev/null

# Alternative brutale : tout arrêter en une commande
$ docker stop $(docker ps -q) 2>/dev/null
$ docker rm $(docker ps -aq) 2>/dev/null
Attention. La commande docker rm $(docker ps -aq) supprime tous les conteneurs de la machine, y compris ceux qui ne font pas partie du TP. Sur un poste partagé ou une VM de production, on préfère toujours supprimer nommément.
Étape 3 — Supprimer les images inutilisées
$ docker image prune        # supprime les images  (dangling)
$ docker image prune -a     # supprime toutes les images non utilisées

Ou sélectivement, image par image :

$ docker rmi devops-monitor:1.0
$ docker rmi alpine:3.20 nginx:1.27-alpine hello-world
Étape 4 — Supprimer le réseau et le volume créés pour le TP
$ docker network rm app-network
$ docker volume rm monitor-logs
Erreur fréquente. Docker refusera de supprimer un volume encore attaché à un conteneur (même arrêté). Il faut d'abord supprimer le conteneur. Même principe pour un réseau : aucun conteneur ne doit y être rattaché.
Étape 5 — La commande "remise à zéro" complète

Pour un nettoyage global en fin de séance : 

$ docker system prune -a --volumes

Cette commande supprime : tous les conteneurs arrêtés, tous les réseaux non utilisés, toutes les images non associées à un conteneur, et tous les volumes anonymes. Docker demande une confirmation explicite — prenez l'habitude de toujours lire ce qu'il s'apprête à faire avant de taper y.

Bonne pratique professionnelle. Mettez en place un petit cron hebdomadaire sur vos hôtes Docker de dev : docker system prune -f --filter "until=168h" (supprime ce qui n'a pas été utilisé depuis 7 jours). Sur un cluster Kubernetes, ce travail est fait automatiquement par le kubelet — vous le verrez en Séance 2.
Mini-synthèse. Savoir créer des conteneurs ne suffit pas : il faut aussi savoir les ranger. Un cluster Kubernetes en production, c'est des milliers de conteneurs lancés et détruits par jour. Sans une bonne hygiène opérationnelle, l'infrastructure devient vite ingérable.

6. Synthèse de fin de séance

Nous avons parcouru, en quatre heures, l'essentiel de ce qu'il faut maîtriser pour aborder sereinement Kubernetes. Prenons un moment pour remettre les idées dans l'ordre et cristalliser ce qui doit absolument être retenu.

6.1 Les grandes idées à retenir

  • Un conteneur n'est pas une VM. Il partage le noyau de l'hôte, démarre en quelques secondes, consomme peu de ressources, et offre une isolation par namespaces et cgroups. La VM reste utile, mais pour déployer des applications, le conteneur est l'outil adapté.
  • Une image est immuable, un conteneur est éphémère. L'image est un modèle qu'on produit une fois et qu'on exécute autant de fois qu'on veut. Le conteneur peut être détruit sans état d'âme : c'est même le but.
  • La configuration vit à l'extérieur de l'image. Variables d'environnement au runtime, pas de secrets dans le Dockerfile, pas d'URL codées en dur. Cette règle prépare directement les ConfigMap et Secret de Kubernetes.
  • Les données persistantes vivent dans des volumes. Le système de fichiers d'un conteneur est jetable. Tout ce qui doit survivre doit être explicitement monté dans un volume.
  • Les conteneurs communiquent via des réseaux nommés. Un réseau bridge personnalisé fournit résolution DNS et isolation. Ces deux concepts seront au cœur des Service Kubernetes.
  • Un Dockerfile est une recette reproductible. Il encapsule tout ce qu'il faut pour que l'application tourne : version de Python, dépendances, code, utilisateur non-root, port exposé, commande de démarrage. Le principe de reproductibilité est garanti par les couches et le cache.

6.2 Ce que vous savez faire maintenant

✓ Compétences acquises

  • Installer et administrer Docker Engine sur Ubuntu Server
  • Manipuler images, conteneurs, volumes et réseaux
  • Écrire un Dockerfile correct pour une application Python
  • Construire une image avec docker build et comprendre les couches
  • Lancer une application conteneurisée avec la bonne configuration
  • Injecter des variables d'environnement
  • Assurer la persistance des données via un volume
  • Faire communiquer deux conteneurs via un réseau bridge personnalisé
  • Nettoyer proprement un environnement Docker

✗ Erreurs conceptuelles à éviter

  • Croire qu'un conteneur est une « mini-VM »
  • Mettre des secrets ou des mots de passe dans le Dockerfile
  • Utiliser le tag :latest en production
  • Compter sur le système de fichiers du conteneur pour conserver des données
  • Utiliser docker exec pour modifier durablement une application (elle sera perdue au prochain docker run)
  • Faire tourner le processus principal en root à l'intérieur du conteneur
  • Utiliser le réseau bridge par défaut alors qu'un bridge personnalisé offre la résolution DNS

6.3 Pourquoi Kubernetes est la suite logique

Revenons au constat que nous avions posé dans la section 2.5 : Docker règle le problème de l'empaquetage d'une application, mais pas celui de l'exploitation à l'échelle. Observez votre VM : vous y avez fait tourner deux ou trois conteneurs, à la main. Imaginez maintenant devoir faire tourner :

  • 50 microservices, chacun en 3 exemplaires pour la redondance ;
  • réparti sur 10 hôtes ESXi ;
  • avec redémarrage automatique en cas de crash ;
  • avec mise à jour progressive sans coupure ;
  • avec montée en charge automatique selon le trafic ;
  • avec supervision centralisée et logs agrégés.

Aucun humain ne peut gérer cela à la main. Il faut un orchestrateur. C'est précisément le rôle de Kubernetes, que nous aborderons dès la séance suivante.

7. Ouverture : de Docker à Kubernetes, puis à Kubeflow/MLOps

La séance 1 vous a donné la brique élémentaire de toute plate-forme cloud-native moderne : le conteneur. La séance 2 va vous apprendre à les orchestrer, et les séances ultérieures à construire un véritable pipeline MLOps par-dessus. Voici la carte mentale de la suite.

7.1 Aperçu des concepts Kubernetes à venir

Concept K8s Rôle Lien avec ce que vous venez d'apprendre
Pod Plus petite unité déployable : un ou plusieurs conteneurs co-localisés. Un pod contient des conteneurs Docker (ou compatibles OCI). C'est l'évolution directe du docker run.
Deployment Gère un ensemble de pods identiques, avec redémarrage automatique et mises à jour progressives. Remplace le docker run --restart=always multiplié par N exemplaires sur plusieurs hôtes.
Service Point d'accès stable (IP + nom DNS) vers un groupe de pods. Extension naturelle du réseau Docker personnalisé que vous avez utilisé en TP 12.
ConfigMap / Secret Externalisation de la configuration et des secrets. Généralisation des options -e et --env-file de Docker.
PersistentVolume Stockage persistant monté dans les pods. Généralisation des volumes Docker, avec support des backends réseau (NFS, Ceph, CSI…).
HorizontalPodAutoscaler Ajuste automatiquement le nombre de pods selon la charge. Ce qu'on ne peut pas faire avec Docker seul : la montée en charge dynamique.
Namespace Cloisonnement logique des ressources dans le cluster. Comparable à l'isolation qu'offre un réseau Docker, mais à l'échelle du cluster.

7.2 Le chemin long : vers Kubeflow et MLOps

Une fois Kubernetes maîtrisé, nous pourrons construire par-dessus une plate-forme spécialisée : Kubeflow. Elle permet de gérer tout le cycle de vie d'un modèle de machine learning — entraînement distribué, versioning, pipelines reproductibles, servir des modèles en production — le tout en conteneurs orchestrés par Kubernetes.

Gardez en tête cette perspective à chaque séance : les notions que nous abordons ne sont pas des compétences isolées, elles sont les marches d'un escalier qui mène vers des architectures cloud-native complètes, utilisées aujourd'hui par Google, Netflix, Spotify, les grandes banques, et de plus en plus par les laboratoires de recherche en IA.

Préparation à la séance 2.
  • Conservez votre VM Ubuntu opérationnelle.
  • Reproduisez sans le support le TP 6 à 9 (écrire un Dockerfile, builder, lancer).
  • Si possible, installez kubectl et testez minikube ou kind en prévision — nous commencerons la Séance 2 par la mise en place d'un petit cluster local.

En un mot

Docker vous a appris à empaqueter une application proprement. Kubernetes vous apprendra à les faire vivre à l'échelle. Kubeflow vous apprendra à faire apprendre des modèles dans cet écosystème. Cette séance était la première marche — la plus importante, parce que toutes les suivantes en dépendent.

8.3 Exercice à faire à la maison

Exercice — Conteneuriser une petite API à deux routes.

Écrivez de zéro une petite application Flask (ou Node.js/Express, au choix) exposant deux routes :

  • GET / : renvoie un message JSON contenant le nom de l'étudiant et le hostname du conteneur.
  • GET /count : renvoie le nombre total d'appels reçus depuis le démarrage du conteneur, persisté dans un fichier.

Consignes :

  1. Écrire le Dockerfile en respectant les bonnes pratiques (image slim, ordre des COPY, utilisateur non-root, EXPOSE).
  2. Builder l'image avec le tag tp1-api:1.0.
  3. Lancer le conteneur avec un volume nommé qui assure la persistance du compteur.
  4. Détruire le conteneur, en recréer un, et vérifier que le compteur reprend là où il s'était arrêté.
  5. Livrer un rapport court (2 pages max) contenant : le code de l'application, le Dockerfile commenté, les commandes utilisées, et une capture d'écran prouvant la persistance.

Ce que j'évaluerai : la propreté du Dockerfile, le bon usage du volume, la clarté des explications, et votre capacité à anticiper au moins un piège (ex. permissions, user non-root qui ne peut écrire dans /app/data, etc.).

Conseil. Cet exercice reprend volontairement tout ce que vous avez vu en TP. Si quelque chose bloque, relisez le support plutôt que de chercher sur Stack Overflow : l'objectif est que vous maîtrisiez la démarche, pas la recette.
