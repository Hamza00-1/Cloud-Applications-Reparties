1. Introduction — Pourquoi cette séance est essentielle

Dans la séance précédente, nous avons découvert Docker. Nous avons appris à créer des images, lancer des conteneurs, écrire un Dockerfile, et comprendre la différence fondamentale entre une machine virtuelle et un conteneur. C'était une première étape essentielle.

Mais posons-nous une question honnête : est-ce que Docker seul suffit pour gérer une vraie plateforme en production ?

Imaginez la situation suivante. Vous avez une application découpée en 15 micro-services, chacun dans son conteneur. Il faut :

  • S'assurer que chaque conteneur tourne en permanence
  • Redémarrer automatiquement ceux qui tombent
  • Gérer la communication entre les services
  • Répartir la charge quand le trafic augmente
  • Exposer certains services à l'extérieur, d'autres non
  • Gérer la configuration et les secrets
  • Stocker des données de façon persistante

Avec Docker seul, tout cela se fait manuellement. C'est là qu'intervient Kubernetes : un système d'orchestration qui automatise la gestion de vos conteneurs à grande échelle.

Et pour nous, Kubernetes n'est pas une fin en soi. C'est la fondation sur laquelle nous allons installer Kubeflow, une plateforme complète de Machine Learning Operations (MLOps). La progression est logique et chaque couche s'appuie sur la précédente :

Aujourd'hui, nous allons franchir le cap. Nous ne resterons pas dans la théorie Kubernetes. L'objectif est clair : comprendre les fondations, puis démarrer réellement l'installation de Kubeflow sur notre cluster de laboratoire. À la fin de cette séance, vous aurez un premier aperçu concret de la plateforme que nous exploiterons dans les prochaines séances.

💡 Rappel important
Kubeflow ne fonctionne pas en dehors de Kubernetes. Chaque notebook, chaque pipeline, chaque interface que nous utiliserons plus tard s'exécute sur le cluster. C'est pour cela que la compréhension — même minimale — de Kubernetes est indispensable avant d'aller plus loin.

≈ 60 min

Cette partie est volontairement concise : l'objectif est de comprendre ce dont nous avons besoin pour le TP, pas de couvrir l'intégralité de Kubernetes. 2.1. Pourquoi Kubernetes après Docker ?

Docker résout un problème fondamental : empaqueter et exécuter une application de façon isolée et reproductible. C'est formidable. Mais dès que l'on dépasse le stade du conteneur unique, de nouveaux défis apparaissent.

DéfiDocker seulAvec Kubernetes
Conteneur qui plante Arrêté, intervention manuelle Redémarrage automatique immédiat
Montée en charge Manuellement lancer d'autres conteneurs Auto-scaling, réplication déclarative
Communication inter-services Réseaux Docker + gestion manuelle Service discovery automatique, DNS interne
Mise à jour sans interruption Stop / Remove / Run (coupure) Rolling update transparent
Répartition sur plusieurs machines Pas de mécanisme natif Scheduling intelligent multi-nœuds
Configuration et secrets Variables d'environnement manuelles ConfigMaps et Secrets gérés par la plateforme
Stockage persistant Volumes Docker (local uniquement) PersistentVolumes, provisionning dynamique

Retenez bien ce point : Docker est l'outil de conteneurisation, Kubernetes est l'outil d'orchestration. Les deux ne sont pas en compétition — ils se complètent. Kubernetes utilise Docker (ou un autre runtime) pour exécuter les conteneurs.

📌 Point clé
Kubernetes ne remplace pas Docker. Il s'appuie dessus. Un cluster Kubernetes exécute des conteneurs via un container runtime (containerd, CRI-O…). Docker reste l'outil avec lequel vous construisez vos images.

2.2. Définition et rôle de Kubernetes

Kubernetes (souvent abrégé K8s) est une plateforme open source d'orchestration de conteneurs, initialement développée par Google et maintenue par la Cloud Native Computing Foundation (CNCF).

Concrètement, Kubernetes joue trois rôles essentiels :

① Un système déclaratif. Vous décrivez l'état souhaité de votre infrastructure dans des fichiers YAML : « Je veux 3 répliques de ce service, avec 512 Mo de mémoire chacune, exposées sur le port 8080 ». Kubernetes se charge de réaliser cet état et de le maintenir en permanence.

② Une base d'exécution pour les plateformes cloud natives. De nombreuses plateformes — Istio (service mesh), Prometheus (monitoring), Argo (CI/CD), et bien sûr Kubeflow — s'installent sur Kubernetes. Elles utilisent ses primitives (Pods, Services, namespaces…) comme briques de base.

③ Un gestionnaire de cycle de vie. Kubernetes surveille vos applications en continu. Si un conteneur tombe, il le redémarre. Si un nœud tombe, il redéploie les conteneurs ailleurs. Il gère les mises à jour progressives, le rollback, la montée en charge.

🎯 Pour notre parcours
Kubernetes est la fondation technique sur laquelle Kubeflow va s'installer et fonctionner. Sans un cluster Kubernetes opérationnel, il n'y a tout simplement pas de Kubeflow possible.

2.3. Architecture générale d'un cluster Kubernetes

Un cluster Kubernetes est un ensemble de machines (physiques ou virtuelles) qui travaillent ensemble. Ce cluster est composé de deux types de nœuds :

Le Control Plane (plan de contrôle) : c'est le cerveau du cluster. Il prend les décisions, planifie les déploiements, surveille l'état global.

Les Worker Nodes (nœuds de travail) : ce sont les machines qui exécutent réellement les conteneurs. C'est sur ces nœuds que vos applications — et les composants Kubeflow — tourneront.

Passons en revue chaque composant du cluster :

ComposantLocalisationRôle
kube-apiserverControl PlanePoint d'entrée unique pour toutes les interactions avec le cluster. Chaque commande kubectl passe par l'API server.
etcdControl PlaneBase de données clé-valeur distribuée qui stocke l'état complet du cluster (configuration, secrets, état des ressources).
kube-schedulerControl PlaneDécide sur quel nœud worker un Pod doit être placé, en tenant compte des ressources disponibles et des contraintes.
controller-managerControl PlaneExécute les boucles de contrôle : vérifie en permanence que l'état réel correspond à l'état déclaré (nombre de répliques, etc.).
kubeletChaque WorkerAgent local qui gère les conteneurs sur son nœud. Il reçoit les instructions du control plane et s'assure que les Pods tournent.
kube-proxyChaque WorkerGère les règles réseau pour acheminer le trafic vers les bons Pods (service discovery, load balancing interne).
containerdChaque WorkerRuntime de conteneurs. C'est le moteur qui lance et gère les conteneurs au niveau du système d'exploitation.
⚠️ Erreur fréquente
Ne confondez pas le control plane avec un simple « serveur d'administration ». Le control plane ne sert pas uniquement à taper des commandes : il contient les composants critiques qui font fonctionner l'ensemble du cluster. Si le control plane tombe, le cluster ne peut plus accepter de nouvelles instructions (même si les conteneurs existants continuent de tourner).

2.4. Objets Kubernetes essentiels avant Kubeflow

Dans Kubernetes, tout est représenté par des objets décrits en YAML. Nous allons passer en revue les objets que vous rencontrerez dès aujourd'hui dans le TP, et que vous reverrez constamment dans Kubeflow.

Pod

Le Pod est l'unité d'exécution la plus petite dans Kubernetes. Un Pod contient un ou plusieurs conteneurs qui partagent le même espace réseau et le même stockage.

En pratique : dans la plupart des cas, un Pod = un conteneur. Kubeflow crée des Pods pour chaque notebook, chaque étape de pipeline, chaque composant d'interface.

YAMLapiVersion: v1
kind: Pod
metadata:
  name: mon-premier-pod
  namespace: default
spec:
  containers:
  - name: nginx
    image: nginx:1.25
    ports:
    - containerPort: 80

Deployment

Un Deployment gère un ensemble de Pods identiques. Il garantit qu'un nombre souhaité de répliques tourne en permanence et permet les mises à jour progressives.

Lien Kubeflow : la plupart des composants Kubeflow (dashboard, contrôleurs, etc.) sont déployés via des Deployments.

Service

Un Service expose un ensemble de Pods sous une adresse réseau stable. Les Pods sont éphémères (ils peuvent être recréés à tout moment), mais le Service maintient un point d'accès permanent.

Lien Kubeflow : le dashboard Kubeflow est accessible via un Service. Les composants internes communiquent entre eux par des Services.

Namespace

Un Namespace est un espace de noms logique qui isole les ressources. C'est comme des « dossiers » dans le cluster.

Lien Kubeflow : chaque utilisateur Kubeflow obtient son propre namespace. Kubeflow installe ses composants dans plusieurs namespaces dédiés (kubeflow, istio-system, auth, etc.).

ConfigMap et Secret

Les ConfigMaps stockent de la configuration non sensible (paramètres, URLs). Les Secrets stockent des données sensibles (mots de passe, tokens) de façon encodée.

Lien Kubeflow : la configuration de Kubeflow repose largement sur des ConfigMaps. Les credentials d'accès sont gérés via des Secrets.

Volume / PersistentVolume

Par défaut, les données dans un conteneur sont éphémères. Les Volumes et PersistentVolumes (PV) permettent de stocker des données de façon durable, indépendamment du cycle de vie du Pod.

Lien Kubeflow : les notebooks Kubeflow stockent les fichiers de travail sur des PersistentVolumes. Sans stockage persistant, vous perdriez tout à chaque redémarrage.

2.5. Pourquoi Kubeflow repose sur ces briques ?

Maintenant que nous connaissons les objets Kubernetes, voyons concrètement comment Kubeflow les utilise. Ce lien est fondamental pour comprendre la suite.

Quand un data scientist ouvre un notebook Jupyter dans Kubeflow, voici ce qui se passe dans le cluster :

  1. Un Pod est créé dans le namespace de l'utilisateur
  2. Ce Pod exécute un conteneur avec une image Jupyter (Python, TensorFlow…)
  3. Un PersistentVolume est monté pour sauvegarder les fichiers
  4. Un Service expose le notebook pour y accéder via le navigateur
  5. Des ConfigMaps et Secrets fournissent la configuration nécessaire

De même, quand un pipeline ML s'exécute, chaque étape du pipeline crée un Pod. Le scheduling, la gestion des ressources, le redémarrage en cas d'erreur — tout cela est géré par Kubernetes.

💡 L'analogie
Pensez à Kubernetes comme au système d'exploitation de votre cluster, et à Kubeflow comme une application installée dessus. Tout comme une application a besoin d'un OS fonctionnel (gestion mémoire, fichiers, réseau), Kubeflow a besoin d'un cluster Kubernetes fonctionnel (pods, services, stockage, namespaces).

3. Commandes kubectl essentielles

kubectl est l'outil en ligne de commande pour interagir avec un cluster Kubernetes. C'est votre interface principale avec le cluster. Familiarisons-nous avec les commandes que nous utiliserons constamment.

CommandeRôleExempleExplication
kubectl version Vérifie la version client et serveur kubectl version --short Confirme que kubectl est installé et communique avec le cluster. Indique la version du client (votre machine) et du serveur (le cluster).
kubectl cluster-info Informations sur le cluster kubectl cluster-info Affiche l'adresse du control plane et du service DNS. Si cette commande échoue, votre connexion au cluster est cassée.
kubectl get nodes Liste les nœuds du cluster kubectl get nodes -o wide Montre chaque nœud, son statut (Ready / NotReady), son rôle (control-plane, worker), sa version. L'option -o wide ajoute l'IP et l'OS.
kubectl get pods Liste les Pods kubectl get pods -A Le flag -A (all namespaces) est crucial : il montre les Pods de tous les espaces, y compris les composants système et Kubeflow.
kubectl get ns Liste les namespaces kubectl get ns Montre les espaces de noms existants. Après l'installation de Kubeflow, vous verrez apparaître de nouveaux namespaces.
kubectl describe Détails d'une ressource kubectl describe pod mon-pod -n default Affiche tous les détails : événements, conditions, configuration, volumes, état des conteneurs. Indispensable pour le debug.
kubectl logs Journaux d'un conteneur kubectl logs mon-pod -n default Affiche la sortie standard du conteneur. Ajoutez -f pour suivre en temps réel (comme tail -f).
kubectl apply -f Applique une configuration YAML kubectl apply -f mon-fichier.yaml Crée ou met à jour les ressources décrites dans le fichier. C'est la commande d'écriture déclarative par excellence.
kubectl delete -f Supprime les ressources d'un YAML kubectl delete -f mon-fichier.yaml Supprime exactement les ressources décrites dans le fichier.
kubectl get svc -A Liste tous les Services kubectl get svc -A Montre tous les points d'accès réseau. Utile pour trouver l'adresse du dashboard Kubeflow.
kubectl get deployments -A Liste tous les Deployments kubectl get deployments -A Montre les Deployments et leur nombre de répliques (READY). Si READY montre 0/1, quelque chose ne va pas.
✅ Conseil pratique
L'option -o wide fonctionne sur presque toutes les commandes get et ajoute des colonnes utiles (IP, nœud, etc.). L'option --watch (ou -w) permet de surveiller les changements en temps réel — très utile pendant l'installation de Kubeflow pour voir les Pods démarrer progressivement.

4. Kubeflow — Vue d'ensemble avant installation

Avant de lancer l'installation, prenons un moment pour comprendre ce que nous allons déployer. Kubeflow n'est pas une simple application : c'est une plateforme composée de plusieurs briques, chacune ayant un rôle dans le cycle de vie du Machine Learning.

4.1. Qu'est-ce que Kubeflow ?

Kubeflow est une plateforme open source conçue pour rendre les workflows de Machine Learning sur Kubernetes simples, portables et scalables. Elle a été initiée par Google et est maintenue par une communauté active.

L'idée fondatrice est la suivante : au lieu de bricoler des scripts, des serveurs Jupyter isolés, des systèmes de suivi d'expériences séparés, Kubeflow offre un environnement intégré qui couvre toutes les étapes du ML — de l'exploration des données au déploiement en production.

4.2. Les grands composants de Kubeflow

5. Architecture de laboratoire proposée

Voyons maintenant comment exploiter notre infrastructure réelle pour créer un cluster de laboratoire. Pour rappel, nous disposons de :

  • 2 hôtes ESXi, chacun avec 32 Go de RAM et 1 To HDD
  • vCenter opérationnel
  • Veeam pour les sauvegardes

Voici l'architecture que nous allons utiliser. C'est une proposition pédagogique adaptée à notre labo — en production, le dimensionnement serait différent et on utiliserait de la haute disponibilité pour le control plane.

VMRôlevCPURAMDisqueHôte ESXiOS
k8s-masterControl Plane K8s + administration kubectl48 Go100 GoESXi-1Ubuntu 22.04 LTS
k8s-worker1Worker Node 1 (exécution des Pods)616 Go200 GoESXi-1Ubuntu 22.04 LTS
k8s-worker2Worker Node 2 (exécution des Pods)616 Go200 GoESXi-2Ubuntu 22.04 LTS
💡 Pourquoi cette répartition ?

Le control plane n'a pas besoin de beaucoup de RAM car il ne lance pas les charges de travail. En revanche, les workers ont besoin de 16 Go chacun car Kubeflow est gourmand : les composants (Istio, dashboard, notebooks, pipelines, etc.) consomment facilement 8 à 12 Go au total. En répartissant les workers sur les deux hôtes ESXi, nous obtenons une certaine résilience matérielle.

⚠️ Rappel
C'est une architecture de laboratoire pédagogique. En production, le control plane serait en haute disponibilité (3 masters), les workers seraient plus nombreux, et le stockage utiliserait des solutions distribuées (Ceph, Longhorn…). Ici, nous privilégions la simplicité pour apprendre.

6. TP guidé — De la vérification du cluster au démarrage de Kubeflow

≈ 160 min

Nous entrons maintenant dans la partie pratique. Suivez chaque étape dans l'ordre. Si vous êtes bloqué, relisez les explications avant de demander de l'aide. L'objectif est de comprendre chaque étape, pas seulement de taper des commandes.

📌 Pré-requis pour le TP

Avant de commencer, assurez-vous que :

  • Les 3 VMs (k8s-master, k8s-worker1, k8s-worker2) sont démarrées et accessibles en SSH
  • Kubernetes est installé sur les 3 VMs (via kubeadm — cette installation a été préparée en amont)

TP 1 — Vérification de l'infrastructure de labo

Objectif : valider que notre environnement est prêt avant toute manipulation Kubernetes

La toute première étape — que l'on oublie souvent — est de vérifier l'infrastructure. Il est inutile de lancer une installation si les VMs n'ont pas assez de ressources ou si la connectivité réseau est cassée.

Étape 1.1 — Vérifier les ressources de chaque VM

Connectez-vous en SSH sur chaque VM et exécutez les commandes suivantes.

Vérifier la RAM disponible :

free -h
total used free shared buff/cache available Mem: 15Gi 1.2Gi 12Gi ... 2.1Gi 13Gi

Vous devez voir environ 8 Go sur le master et 16 Go sur chaque worker. Si la RAM disponible est significativement inférieure, vérifiez la configuration de la VM dans vCenter.

Vérifier les CPU :

nproc
6

Vérifier l'espace disque :

df -h /
Filesystem Size Used Avail Use% Mounted on /dev/sda2 196G 12G 175G 7% /

Kubeflow télécharge beaucoup d'images Docker. Vérifiez qu'il reste au moins 80 Go disponibles sur chaque worker.

Étape 1.2 — Vérifier la connectivité réseau

Depuis le master, vérifiez que vous pouvez atteindre les deux workers :

ping -c 3 k8s-worker1
ping -c 3 k8s-worker2

Si le DNS ne résout pas les noms, utilisez les adresses IP directement. Vérifiez aussi la résolution DNS externe :

ping -c 3 google.com
🚫 Erreur fréquente
Si la résolution DNS externe échoue, l'installation de Kubeflow ne pourra pas télécharger les images Docker. Vérifiez la configuration DNS dans /etc/resolv.conf et assurez-vous que les VMs ont accès à Internet.

Étape 1.3 — Vérifier que le runtime de conteneurs fonctionne

Sur chaque nœud, vérifiez que containerd est actif :

sudo systemctl status containerd
● containerd.service - containerd container runtime Loaded: loaded (/lib/systemd/system/containerd.service; enabled) Active: active (running) since ...

🔑 Pourquoi cette étape est indispensable

Une installation de Kubeflow échouera silencieusement si le cluster manque de ressources, si la connectivité est cassée, ou si le runtime de conteneurs ne fonctionne pas. Ces vérifications prennent 5 minutes et vous éviteront des heures de debugging.

TP 2 — Vérification du cluster Kubernetes

Objectif : confirmer que le cluster est sain et prêt à accueillir Kubeflow

Maintenant que l'infrastructure est validée, vérifions que le cluster Kubernetes lui-même est opérationnel.

Étape 2.1 — Vérifier les nœuds

kubectl get nodes -o wide
NAME STATUS ROLES AGE VERSION INTERNAL-IP OS-IMAGE k8s-master Ready control-plane 2d v1.28.4 192.168.1.10 Ubuntu 22.04 LTS k8s-worker1 Ready <none> 2d v1.28.4 192.168.1.11 Ubuntu 22.04 LTS k8s-worker2 Ready <none> 2d v1.28.4 192.168.1.12 Ubuntu 22.04 LTS

Trois points à vérifier :

  1. STATUS = Ready pour les 3 nœuds. Si un nœud est NotReady, la kubelet ne communique pas correctement avec le control plane.
  2. VERSION identique sur tous les nœuds. Des versions différentes peuvent causer des problèmes de compatibilité.
  3. ROLES : le master doit avoir control-plane, les workers sont sans rôle explicite ou marqués worker.

Étape 2.2 — Vérifier les composants système

kubectl get pods -n kube-system
NAME READY STATUS RESTARTS AGE coredns-5dd5756b68-xxxxx 1/1 Running 0 2d coredns-5dd5756b68-yyyyy 1/1 Running 0 2d etcd-k8s-master 1/1 Running 0 2d kube-apiserver-k8s-master 1/1 Running 0 2d kube-controller-manager-k8s-master 1/1 Running 0 2d kube-proxy-xxxxx 1/1 Running 0 2d kube-proxy-yyyyy 1/1 Running 0 2d kube-proxy-zzzzz 1/1 Running 0 2d kube-scheduler-k8s-master 1/1 Running 0 2d calico-node-xxxxx 1/1 Running 0 2d ...

Tous les Pods du namespace kube-system doivent être en Running. Retrouvons ce que nous avons vu en théorie :

  • kube-apiserver, etcd, kube-scheduler, kube-controller-manager → composants du control plane
  • kube-proxy → un par nœud, pour le routage réseau
  • coredns → DNS interne du cluster
  • calico-node (ou votre plugin réseau) → gestion du réseau inter-pods

Étape 2.3 — Vérifier les namespaces existants

kubectl get ns
NAME STATUS AGE default Active 2d kube-node-lease Active 2d kube-public Active 2d kube-system Active 2d

À ce stade, vous ne devez voir que les namespaces par défaut de Kubernetes. Après l'installation de Kubeflow, cette liste sera beaucoup plus longue.

Étape 2.4 — Vérifier les informations du cluster

kubectl cluster-info
Kubernetes control plane is running at https://192.168.1.10:6443 CoreDNS is running at https://192.168.1.10:6443/api/v1/namespaces/kube-system/services/kube-dns:dns/proxy
✅ Point de validation
Si toutes les commandes ci-dessus retournent des résultats corrects, votre cluster est sain et prêt pour la suite. Si quelque chose ne va pas, ne continuez pas — résolvez le problème d'abord.

TP 3 — Rappel rapide : créer une ressource simple Kubernetes

Objectif : ancrer la logique Pod / Service avant Kubeflow (exercice court)

Avant de plonger dans Kubeflow, faisons un exercice très rapide pour s'assurer que vous maîtrisez le cycle de base : créer un Pod, l'exposer via un Service, vérifier que tout fonctionne.

Étape 3.1 — Créer un Deployment nginx

Créez un fichier test-nginx.yaml sur le master :

YAMLapiVersion: apps/v1
kind: Deployment
metadata:
  name: test-nginx
  namespace: default
spec:
  replicas: 2
  selector:
    matchLabels:
      app: test-nginx
  template:
    metadata:
      labels:
        app: test-nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: test-nginx-svc
  namespace: default
spec:
  type: NodePort
  selector:
    app: test-nginx
  ports:
  - port: 80
    targetPort: 80
    nodePort: 30080

Cet exercice était juste un rappel. Nettoyons :

TP 5 — Démarrage de l'installation de Kubeflow

Objectif : lancer le déploiement complet de Kubeflow — c'est le cœur du TP

Nous y sommes. C'est le moment le plus important de la séance. Nous allons lancer l'installation de l'ensemble de la plateforme Kubeflow sur notre cluster.

Étape 5.1 — Comprendre ce que nous allons déployer

L'installation via les manifests officiels déploie l'intégralité de Kubeflow en une seule commande. Cela inclut :

  • Istio — service mesh pour la gestion du trafic et la sécurité réseau
  • cert-manager — gestion automatique des certificats TLS
  • Dex — authentification et gestion d'identité (OIDC)
  • OIDC AuthService — middleware d'authentification
  • Kubeflow namespace — espace dédié aux composants
  • Central Dashboard — interface web principale
  • Notebooks Controller — gestion des notebooks Jupyter
  • Kubeflow Pipelines — orchestration de workflows ML
  • KServe — déploiement de modèles (model serving)
  • Katib — optimisation d'hyperparamètres
  • Training Operator — gestion des jobs d'entraînement distribués
  • Profils / KFAM — gestion multi-utilisateurs
  • Volumes Web App — interface de gestion du stockage

C'est beaucoup de composants, et c'est normal. Kubeflow est une plateforme complète. Nous n'utiliserons pas tout immédiatement, mais tout sera en place pour les prochaines séances.

Étape 5.2 — Lancer l'installation

Depuis le répertoire ~/manifests, exécutez :

cd ~/manifests
BASH# Installation complète — cette commande peut prendre 10 à 20 minutes
while ! kustomize build example | kubectl apply -f -; do
  echo "Retrying to apply resources..."
  sleep 10
done
⚠️ Pourquoi une boucle while ?

L'installation crée des ressources dans un ordre spécifique. Certaines ressources dépendent d'autres (CRDs — Custom Resource Definitions) qui doivent être créées en premier. La boucle while ré-applique automatiquement les manifests jusqu'à ce que toutes les dépendances soient satisfaites. C'est normal de voir des erreurs temporaires lors des premières itérations.

Pendant que l'installation tourne, ouvrez un second terminal SSH sur le master et surveillez l'évolution :

watch -n 5 "kubectl get pods -A | grep -v Running | grep -v Completed"

Cette commande rafraîchit toutes les 5 secondes et affiche uniquement les Pods qui ne sont pas encore en Running. Au début, vous verrez beaucoup de Pods en ContainerCreating ou Init. Progressivement, ils passeront en Running.

Étape 5.3 — Comprendre ce qui se passe pendant l'installation

Pendant que les Pods démarrent, expliquons les phases de l'installation :

Phase 1 — CRDs et namespaces (premières secondes)

Kubernetes crée les Custom Resource Definitions (nouveaux types d'objets) et les namespaces nécessaires. C'est la fondation sur laquelle tout le reste repose.

Phase 2 — Composants infrastructure (1-5 min)

Istio, cert-manager et Dex démarrent. Ce sont les « services de support » dont Kubeflow a besoin. Istio en particulier est complexe et prend du temps.

Phase 3 — Composants Kubeflow (5-15 min)

Les composants applicatifs démarrent : dashboard, notebooks controller, pipelines, KServe, etc. Ils téléchargent leurs images Docker, s'initialisent, et se connectent aux services d'infrastructure.

Phase 4 — Stabilisation (15-20 min)

Tous les Pods atteignent l'état Running. Les contrôleurs vérifient que tout est fonctionnel. Les readiness probes confirment que les services sont prêts.

🚫 Problèmes fréquents pendant l'installation
  • ImagePullBackOff — le cluster n'arrive pas à télécharger une image Docker. Vérifiez la connectivité Internet et le DNS.
  • Pending prolongé — pas assez de ressources (RAM, CPU). Vérifiez avec kubectl describe pod <nom> -n <namespace>.
  • CrashLoopBackOff — le conteneur démarre et plante en boucle. Consultez les logs avec kubectl logs <pod> -n <namespace>.
  • Erreurs de dépendances — certaines ressources ne trouvent pas leurs CRDs. La boucle while résout généralement ce problème. Si ça persiste, relancez la commande.

7. Lire une installation Kubeflow avec recul

Après avoir installé et observé Kubeflow, il est essentiel de savoir comment lire ce type d'installation sans se perdre. Voici une méthodologie que vous pourrez réutiliser pour toute plateforme déployée sur Kubernetes.

7.1. Commencer par les namespaces

Les namespaces vous donnent une carte de haut niveau. Chaque namespace correspond à un domaine fonctionnel. kubectl get ns est toujours votre premier réflexe.

7.2. Explorer les Pods par namespace

Dans chaque namespace, regardez les Pods (kubectl get pods -n <ns>). Le nom du Pod contient souvent le nom du composant. L'état READY et STATUS vous indique la santé.

7.3. Identifier les Services

Les Services (kubectl get svc -n <ns>) vous montrent les points d'accès. Le type (ClusterIP, NodePort, LoadBalancer) indique si le service est interne ou externe.

7.4. Repérer les Deployments et contrôleurs

Les Deployments gèrent les Pods. kubectl get deploy -n <ns> vous montre quoi est géré et combien de répliques sont actives.

7.5. Comprendre le stockage

Les PersistentVolumeClaims (kubectl get pvc -n <ns>) montrent quels composants utilisent du stockage persistant et combien.

7.6. Identifier l'authentification et le routage

Istio gère le routage réseau via des VirtualServices et Gateways. Dex gère l'authentification. Ces composants sont « invisibles » au quotidien mais essentiels.

🔑 Méthode de lecture en 4 étapes

① Namespaces → vue d'ensemble / ② Pods → composants actifs / ③ Services → points d'accès / ④ PVC → stockage. Cette séquence fonctionne pour n'importe quelle plateforme sur Kubernetes, pas seulement Kubeflow.

8. Synthèse de fin de séance

Ce qu'il faut retenir sur Kubernetes

Kubernetes est un orchestrateur de conteneurs qui automatise le déploiement, la gestion et la mise à l'échelle. Son architecture repose sur un control plane (cerveau) et des workers (exécution). Les objets fondamentaux — Pod, Deployment, Service, Namespace, ConfigMap, Secret, PersistentVolume — sont les briques avec lesquelles tout est construit, y compris Kubeflow.

Ce qu'il faut retenir sur Kubeflow

Kubeflow est une plateforme MLOps complète qui s'installe sur Kubernetes. Elle est composée de multiples briques (notebooks, pipelines, serving, monitoring) déployées comme des Pods et Services Kubernetes. Comprendre Kubernetes est indispensable pour comprendre et opérer Kubeflow.
