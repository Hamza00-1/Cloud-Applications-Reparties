1. Introduction

Présentation du TP

Ce travail pratique constitue la suite logique de vos installations de VMware ESXi et Proxmox VE réalisées lors des séances précédentes. Vous disposez désormais de deux hyperviseurs fonctionnels, déployés en nested virtualization sur VMware Workstation. L'objectif est maintenant de prendre en main, comparer et sécuriser ces deux plateformes dans une perspective de cybersécurité.

Vous allez apprendre à administrer concrètement un hyperviseur — créer des machines virtuelles, gérer le réseau, le stockage, les snapshots, les utilisateurs — tout en gardant un regard critique de professionnel de la sécurité : surface d'attaque, moindre privilège, journalisation, segmentation réseau.

🎯 Objectifs d'apprentissage
  • Comprendre l'architecture interne de ESXi et Proxmox VE
  • Administrer les ressources (CPU, RAM, stockage, réseau) sur chaque plateforme
  • Créer, configurer et gérer des machines virtuelles
  • Appliquer des mesures de durcissement sécurité (hardening)
  • Lire et exploiter les journaux d'événements et d'accès
  • Gérer les utilisateurs, rôles et permissions (RBAC)
  • Comparer les deux plateformes dans un contexte académique, SOC et infrastructure cybersécurité

2. Rappels d'architecture & concepts

2.1 Hyperviseur Type 1 vs Type 2

Un hyperviseur est un logiciel (ou firmware) qui permet de créer et d'exécuter des machines virtuelles. On distingue deux types fondamentaux :

Type 1 — Bare-metal

S'installe directement sur le matériel, sans OS intermédiaire. Il est lui-même le système d'exploitation minimal qui gère l'accès au hardware.

Exemples : VMware ESXi, Proxmox VE, Microsoft Hyper-V (rôle serveur), Citrix Hypervisor.

Avantages : performances optimales, accès direct aux ressources, meilleure isolation.

Type 2 — Hosted

S'installe par-dessus un système d'exploitation existant (Windows, Linux, macOS). L'hyperviseur est une application comme une autre.

Exemples : VMware Workstation, VirtualBox, Parallels Desktop.

Avantages : facilité d'installation, idéal pour le développement et le test.

🔒 Point sécurité — Isolation
Un hyperviseur Type 1 offre une meilleure isolation car il n'y a pas d'OS hôte pouvant être compromis. La surface d'attaque est réduite : il n'y a pas de navigateur, pas de services inutiles. C'est pour cette raison que les environnements de production utilisent des hyperviseurs bare-metal.

2.2 VM, Templates, Snapshots

ConceptDéfinitionAnalogie
Machine Virtuelle (VM) Un environnement isolé qui simule un ordinateur complet (CPU, RAM, disque, réseau). Elle exécute son propre OS (« guest OS »). Un ordinateur « logiciel » dans un ordinateur physique.
Template Une VM « modèle » préconfigurée, figée, à partir de laquelle on peut déployer rapidement de nouvelles VMs identiques. La template n'est pas démarrable directement. Un moule à gâteau : on ne le mange pas, mais on s'en sert pour produire des gâteaux identiques.
Snapshot Une « photographie » de l'état d'une VM à un instant T (disque + mémoire optionnellement). Permet de revenir en arrière en cas d'erreur. Un point de sauvegarde dans un jeu vidéo.
⚠️ Attention — Snapshots ≠ Sauvegardes
Un snapshot n'est pas une sauvegarde ! Il est dépendant du disque original. Si le disque est corrompu, le snapshot est inutilisable. De plus, accumuler des snapshots dégrade les performances. En production, utilisez des sauvegardes complètes (backup).

2.3 Stockage : Datastore, ISO, Thin/Thick provisioning

Le stockage est un élément critique de toute infrastructure virtualisée. Voici les concepts clés :

ConceptESXiProxmox
Datastore / Storage Espace de stockage formaté en VMFS (ou NFS). Contient les fichiers des VMs (.vmdk, .vmx, etc.) Pools de stockage : local (répertoire), local-lvm (LVM thin), NFS, CIFS, ZFS, Ceph…
ISO Storage Dossier dans le datastore où sont uploadées les images ISO Storage de type « ISO image » (généralement local)
Thin provisioning Le fichier .vmdk ne fait que la taille réellement utilisée et grossit à la demande Comportement par défaut avec LVM-thin ou qcow2
Thick provisioning L'espace disque total est alloué immédiatement (meilleure performance, plus d'espace consommé) Format raw avec allocation fixe
🔒 Point sécurité — Thick Eager Zeroed
En environnement sensible, le format Thick Provision Eager Zeroed (ESXi) est recommandé. Il écrit des zéros sur tout l'espace alloué, empêchant la récupération de données anciennes. Cela protège contre la fuite de données résiduelles (data remanence).

2.4 Réseau : vSwitch, Bridge, NIC virtuelle, VLAN

La virtualisation réseau est fondamentale pour l'isolation et la segmentation.

ESXi — Virtual Switch (vSwitch)

ESXi utilise des vSwitchs — des commutateurs virtuels internes qui connectent les vNICs des VMs aux NICs physiques (uplinks) de l'hôte.

  • vSwitch standard : configuré localement sur chaque hôte
  • Port Group : groupe de ports sur un vSwitch, avec un label et un VLAN ID optionnel
  • VMkernel port : port spécial pour le trafic de gestion, vMotion, etc.

Proxmox — Bridge Linux

Proxmox utilise les bridges Linux natifs (ex: vmbr0). Un bridge est l'équivalent d'un switch virtuel.

  • vmbr0 : bridge par défaut, relié à l'interface physique
  • On peut créer des bridges supplémentaires (vmbr1, vmbr2…) pour la segmentation
  • Support natif des VLAN tags et des bonds
  • Configuration dans /etc/network/interfaces

3.3 Réseau : vSwitch & Port Groups

Le réseau virtuel ESXi repose sur les Virtual Switches. Par défaut, un vSwitch0 est créé avec un port group VM Network et un port Management Network.

1
Observer la configuration réseau existante

Allez dans Networking → Virtual switches. Vous devriez voir vSwitch0 avec un uplink (vmnic0). Cliquez dessus pour voir sa configuration : MTU, nombre de ports, politique de sécurité.

2
Observer les Port Groups

Allez dans Networking → Port groups. Vous verrez au minimum :

  • Management Network — utilisé pour l'interface d'administration
  • VM Network — port group par défaut pour les VMs
3
Créer un nouveau Port Group (réseau isolé)

Cliquez sur Add port group :

  • Name : Lab-Isolated
  • VLAN ID : 0 (aucun, ou choisissez un ID si vous souhaitez tester le tagging)
  • Virtual switch : vSwitch0

Cliquez Add. Ce port group servira pour le mini-lab isolé de la Partie D.

🔒 Sécurité — Politique de sécurité des Port Groups
Par défaut, ESXi désactive le mode promiscuous, les MAC address changes et le forged transmits sur chaque port group. Ce sont des mesures de sécurité importantes :
  • Promiscuous mode = Reject : empêche une VM de capturer le trafic des autres VMs
  • MAC address changes = Reject : empêche une VM de changer son adresse MAC (anti-spoofing)
  • Forged transmits = Reject : bloque les trames avec une MAC source falsifiée
Ne modifiez ces paramètres qu'avec une raison légitime (ex: VM de monitoring réseau, nested virtualization).
✅ Vérification
Vérifiez que le port group Lab-Isolated apparaît dans la liste. Capturez l'écran pour votre rapport.

  • Name : Avant-installation
  • Description : État avant installation de l'OS
  • Include memory : Oui (optionnel, consomme plus d'espace)

Allez dans Actions ��� Snapshots → Manage snapshots. Vous devez voir votre snapshot dans l'arbre.

⚠️ Attention — Clonage sur ESXi standalone
L'ESXi sans vCenter ne dispose pas de fonction de clonage native dans l'interface web. Le clonage nécessite vCenter Server ou l'utilisation de commandes vmkfstools en CLI. En labo, on utilisera les snapshots ou l'export/import OVF comme alternative.

🔒 Sécurité — Centralisation des logs
En environnement de production ou SOC, les logs ESXi doivent être envoyés vers un serveur syslog distant (SIEM). Cela empêche un attaquant qui compromet l'hôte de supprimer ses traces. Configuration : Manage → System → Advanced Settings → Syslog.global.logHost.

3.7 Comptes, permissions & durcissement

1
Gestion des comptes

Allez dans Manage → Security & Users → Users. Par défaut, seul root existe. Vous pouvez créer un utilisateur supplémentaire avec un rôle limité :

  • Cliquez Add user
  • Username : auditeur
  • Password : (complexe, minimum 8 caractères avec majuscule, chiffre, caractère spécial)
  • Role : Read-Only
2
Vérifier les services exposés

Allez dans Manage → Services. Vérifiez quels services sont actifs :

ServiceRecommandation sécurité
SSH (TSM-SSH)⚠️ Désactiver sauf besoin ponctuel. Chaque activation doit être justifiée et tracée.
ESXi Shell (TSM)⚠️ Désactiver. L'accès shell local est un risque si l'accès physique n'est pas contrôlé.
DCUILaisser actif (nécessaire pour la console directe).
NTP Daemon✅ Activer et configurer. La synchronisation temporelle est critique pour les logs et l'audit.
🔒 Checklist de durcissement ESXi
  • ✅ Mot de passe root complexe (> 12 caractères)
  • ✅ Créer des comptes nominatifs avec rôles limités
  • ✅ Désactiver SSH et ESXi Shell quand non nécessaires
  • ✅ Configurer un timeout pour le shell : UserVars.ESXiShellTimeOut = 600
  • ✅ Activer le lockdown mode (si vCenter disponible)
  • ✅ Configurer un serveur NTP
  • ✅ Envoyer les logs vers un syslog distant
  • ✅ Limiter l'accès réseau à l'interface de management (firewall hôte)
  • ✅ Mettre à jour ESXi régulièrement (patchs de sécurité)
  • ✅ Vérifier les politiques de sécurité des port groups (promiscuous, MAC changes, forged transmits)

Si vous avez activé SSH (temporairement et pour ce TP uniquement), voici des commandes utiles :

4.3 Réseau : bridges Linux

Cliquez Create → Linux Bridge :

  • Name : vmbr1
  • IPv4/CIDR : 10.10.10.1/24 (réseau isolé)
  • Bridge ports : (laisser vide = pas d'uplink physique = réseau isolé)

🔒 Sécurité — Segmentation réseau
Créer un bridge sans uplink physique (bridge ports vide) crée un réseau totalement isolé du réseau externe. Les VMs connectées à vmbr1 ne pourront communiquer qu'entre elles. C'est idéal pour un lab de pentest ou un environnement d'analyse de malware.

4.6 Conteneurs LXC (optionnel)

Proxmox supporte nativement les conteneurs Linux (LXC) en plus des VMs. Un conteneur partage le noyau de l'hôte, consomme moins de ressources, mais offre une isolation moindre qu'une VM complète.

VM (KVM/QEMU)

  • Noyau propre (guest kernel)
  • Émulation matérielle complète
  • Isolation forte
  • Supporte tout OS
  • Plus de ressources consommées

Conteneur LXC

  • Partage le noyau de l'hôte
  • Pas d'émulation matérielle
  • Isolation par namespaces/cgroups
  • Linux uniquement
  • Très léger et rapide
1
Télécharger un template CT

Allez dans local (storage) → CT Templates → Templates. Téléchargez un template (ex: debian-12-standard).

2
Créer le conteneur

Cliquez Create CT. Paramètres : ID 200, hostname ct-test, password, template Debian, 512 MB RAM, 4 GB disk, bridge vmbr0 ou vmbr1.

🔒 Sécurité — Conteneurs non privilégiés
Proxmox crée par défaut des conteneurs non privilégiés (unprivileged). Cela signifie que le root dans le conteneur est mappé sur un utilisateur non-root de l'hôte, limitant considérablement les risques d'évasion de conteneur. N'activez jamais le mode « privileged » sans raison impérative.

7. Partie E — Troubleshooting guidé

🎯 Objectif
Savoir diagnostiquer et résoudre les pannes les plus fréquentes rencontrées en environnement virtualisé, particulièrement en nested virtualization.

🔴 Problème 1 : La VM ne démarre pas

Symptôme La VM reste en état « powered off » ou affiche une erreur au démarrage.

Cause probable

  • VT-x/AMD-V non exposé à la VM hôte (nested virtualization non activée)
  • Mémoire insuffisante sur l'hôte
  • Fichier disque corrompu ou manquant
  • ISO mal monté ou absent

Diagnostic & correction

  • ESXi : Consultez Monitor → Events pour l'erreur exacte. Vérifiez dans VMware Workstation que « Virtualize Intel VT-x/EPT » est coché dans les paramètres processeur de la VM ESXi.
  • Proxmox : Vérifiez les logs : journalctl -u pve-qemu-server -n 50. Si l'erreur mentionne KVM, vérifiez cat /proc/cpuinfo | grep vmx.

🔴 Problème 2 : Pas de réseau dans la VM

Symptôme La VM n'obtient pas d'IP ou ne ping rien.

Cause probable

  • VM connectée au mauvais port group / bridge
  • IP non configurée (pas de DHCP sur le réseau isolé)
  • Driver réseau non installé dans le guest (VirtIO sans pilote)
  • Bridge mal configuré côté Proxmox (vmbr1 pas appliqué)

Correction

  • Vérifiez le port group / bridge assigné à la VM
  • Configurez une IP statique si pas de DHCP
  • Pour VirtIO sous Windows : installez les virtio-win drivers
  • Proxmox : brctl show pour vérifier que le bridge existe et que l'interface de la VM y est connectée

🔴 Problème 3 : Interface web inaccessible

Symptôme Le navigateur affiche « connection refused » ou timeout.

Cause probable

  • Mauvaise IP ou mauvais port (oubli du :8006 pour Proxmox)
  • Le service web est arrêté
  • Firewall bloquant sur le poste ou sur l'hôte
  • Adaptateur réseau de VMware Workstation en mode incorrect (NAT vs Host-Only)

Correction

  • ESXi : vérifiez l'IP sur la DCUI. Testez ping <IP_ESXi> depuis le poste
  • Proxmox : connectez-vous à la console Workstation, vérifiez ip a et systemctl status pveproxy
  • Vérifiez le mode réseau de VMware Workstation (NAT donne accès depuis l'hôte)

🔴 Problème 4 : Datastore / Storage plein

Symptôme Impossible de créer une VM, erreur d'espace disque.

Cause probable

  • Snapshots accumulés consommant l'espace
  • Sauvegardes volumineuses stockées localement
  • Disque de la VM hôte trop petit

Correction

  • ESXi : esxcli storage filesystem list — Supprimez les snapshots inutiles, nettoyez les ISO
  • Proxmox : pvesm status, df -h — Supprimez les anciens backups (/var/lib/vz/dump/)
  • Augmentez la taille du disque de la VM hôte dans VMware Workstation puis étendez la partition

🔴 Problème 5 : Lenteur extrême en nested virtualization

Symptôme Tout est extrêmement lent : boot, console, opérations.

Cause probable

  • VT-x/AMD-V non activé dans le BIOS de la machine physique
  • RAM insuffisante allouée à la VM hôte (ESXi/Proxmox)
  • Swap excessive sur l'hôte physique
  • Antivirus sur le poste hôte analysant les fichiers VM

Correction

  • Vérifiez le BIOS physique : VT-x / AMD-V / SVM doit être Enabled
  • Allouez au minimum 4 GB à la VM ESXi/Proxmox (8 GB idéal)
  • Réduisez les ressources des VMs imbriquées (1 vCPU, 512-1024 MB)
  • Excluez le dossier VMware Workstation de l'antivirus
  • Utilisez un SSD sur le poste physique si possible

🔒 Message final — La sécurité est un processus, pas un produit

La virtualisation est un pilier de l'infrastructure moderne. En tant que futurs professionnels de la cybersécurité, vous devez non seulement savoir administrer ces plateformes, mais surtout les sécuriser, les auditer et les surveiller.

Retenez ces principes fondamentaux :

  • Moindre privilège — N'accordez que les droits nécessaires
  • Défense en profondeur — Ne reposez pas sur une seule mesure de sécurité
  • Journalisation — Ce qui n'est pas loggé n'est pas auditable
  • Segmentation — Isolez les réseaux et les rôles
  • Mises à jour — Un hyperviseur non patché est une porte ouverte
  • Sauvegardes — Testez régulièrement la restauration
