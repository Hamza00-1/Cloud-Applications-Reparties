A — Introduction

Résumé du TP précédent et continuité

Lors du premier TP, vous avez mis en place un environnement de virtualisation imbriquée (nested virtualization) sur VMware Workstation. Vous avez installé deux hyperviseurs de type 1 en tant que machines virtuelles : VMware ESXi (hyperviseur propriétaire VMware) et Proxmox VE (hyperviseur libre basé sur KVM/QEMU). Chaque hyperviseur a été configuré avec son réseau de management, un datastore local et un accès à l'interface web d'administration.

Ce second TP constitue la suite logique : vous allez maintenant passer de l'administration d'un hôte isolé à une administration centralisée. Côté VMware, cela signifie déployer et utiliser vCenter Server pour fédérer votre hôte ESXi dans une structure Datacenter/Cluster. Côté Proxmox, cela signifie explorer en profondeur la vue Datacenter qui offre des fonctionnalités équivalentes (gestion centralisée des nœuds, du stockage, des permissions, du firewall et des sauvegardes).

L'angle cybersécurité sera omniprésent : chaque manipulation intégrera des considérations de durcissement, de contrôle d'accès (RBAC), de journalisation et de segmentation réseau.

B — Partie théorique (Cours détaillé)

B.1 — Hyperviseur et couche management : ESXi seul vs vCenter

Un hyperviseur de type 1 comme ESXi fonctionne de manière autonome : il gère les VMs, le réseau virtuel et le stockage sur un seul hôte physique. L'interface web du Host Client ESXi permet d'administrer cet hôte unique. Cependant, dans un environnement de production, on déploie généralement plusieurs hôtes ESXi. Les administrer un par un devient rapidement ingérable : pas de vue consolidée, pas de migration de VMs entre hôtes, pas de politiques de sécurité cohérentes.

vCenter Server est la couche de management centralisée de VMware. C'est une appliance (VCSA — vCenter Server Appliance) qui se connecte à un ou plusieurs hôtes ESXi et fournit un point d'administration unique. vCenter apporte des fonctionnalités impossibles sans lui : vMotion (migration à chaud), DRS (répartition automatique de charge), HA (haute disponibilité), gestion centralisée des permissions et des réseaux distribués.

Figure B.1 — Architecture vCenter : administration centralisée de plusieurs hôtes ESXi
A retenir
ESXi seul = gestion d'un hôte unique. vCenter = gestion centralisée de N hôtes avec fonctionnalités avancées (HA, DRS, vMotion, RBAC global, réseaux distribués). Même avec un seul hôte, vCenter apporte la structure organisationnelle (Datacenter, Cluster, Folders) et un RBAC granulaire indispensable en cybersécurité.
Micro-TP (5 min) — Réflexion

Connectez-vous à l'interface web de votre ESXi (https://<IP_ESXi>). Identifiez dans le menu de gauche les objets gérables (VMs, stockage, réseau). Notez trois limitations par rapport à ce que vous attendez d'une gestion multi-hôtes (permissions fines, migration, vue consolidée…).

B.2 — Modèle objet vCenter

vCenter organise l'infrastructure en une hiérarchie d'objets logiques. Comprendre cette hiérarchie est fondamental pour structurer un environnement et appliquer des permissions de sécurité de manière cohérente.

Figure B.2 — Hiérarchie des objets dans vCenter

A retenir
Les permissions dans vCenter se propagent du parent vers les enfants dans la hiérarchie. Un rôle attribué au Datacenter s'applique à tous les clusters, hôtes et VMs en dessous — sauf si explicitement remplacé. Cette propagation est un levier puissant mais aussi un risque si mal maîtrisée.

Sur papier ou outil de dessin, esquissez la hiérarchie vCenter que vous allez créer dans ce TP : un Datacenter nommé CYBER-DC, un Cluster CYBER-CLUSTER, votre hôte ESXi, deux folders (Lab-VMs et Templates). Annotez où vous appliqueriez des permissions différenciées.

B.3 — Réseau vSphere

Le réseau virtuel dans vSphere repose sur des commutateurs virtuels (vSwitch). Chaque hôte ESXi possède au moins un vSwitch standard (vSS) créé à l'installation, relié à une carte réseau physique (ou virtuelle en nested). Les Port Groups sont des groupes de ports sur un vSwitch auxquels on assigne des VMs — chaque Port Group peut être associé à un VLAN ID pour segmenter le trafic.

Figure B.3 — Architecture d'un vSwitch standard avec Port Groups
Info — vSwitch Standard vs Distribué (vDS)

Le vSwitch Standard (vSS) est configuré individuellement sur chaque hôte. Le vSwitch Distribué (vDS), disponible uniquement via vCenter (licence Enterprise Plus), offre une configuration réseau centralisée sur plusieurs hôtes. Dans ce TP nested, nous utiliserons le vSS, mais il est important de connaître le vDS pour les environnements de production.

ConceptDescription
Port GroupGroupe de ports assignable aux VMs ou VMkernel, avec VLAN ID optionnel
VMkernelInterface réseau de l'hôte ESXi lui-même (management, vMotion, stockage IP)
VLAN taggingLe Port Group peut étiqueter le trafic avec un VLAN ID (802.1Q) pour la segmentation
UplinkCarte réseau physique (vmnic) reliant le vSwitch au réseau externe
Sécurité — Segmentation réseau

En cybersécurité, séparer le réseau de management (accès admin ESXi/vCenter) du réseau des VMs est une mesure fondamentale. Un attaquant compromettant une VM ne doit pas pouvoir atteindre l'interface de management de l'hyperviseur. Utilisez des Port Groups distincts et, idéalement, des VLANs séparés.

Micro-TP (5 min) — Inventaire réseau ESXi

Dans l'interface ESXi Host Client, naviguez vers Networking → Virtual switches. Identifiez le vSwitch existant, ses port groups et la carte réseau associée (vmnic0). Notez ces informations — elles serviront dans la partie TP.

B.4 — Stockage vSphere

Le stockage dans vSphere s'organise autour du concept de Datastore : un volume de stockage présenté aux hôtes ESXi, sur lequel sont stockés les fichiers des VMs (disques VMDK, fichiers de configuration, swap, snapshots), les images ISO et les templates. Un datastore peut reposer sur un stockage local (disque de l'hôte), NFS, iSCSI ou Fibre Channel.

Type de provisionnementDescriptionImplications
Thick Provisioned (Eager Zeroed)Espace disque entièrement alloué et initialisé à zéro immédiatementMeilleures performances, espace réservé — recommandé en production sécurisée (pas de données résiduelles)
Thick Provisioned (Lazy Zeroed)Espace alloué mais initialisé à la demandeRisque théorique de lecture de données résiduelles sur le stockage
Thin ProvisionedEspace alloué à la demande (le fichier VMDK grossit)Économie d'espace, mais risque de surprovisionnement — inadapté si le datastore se remplit
Sécurité — Stockage

Le provisionnement Eager Zeroed est préféré dans les environnements sensibles car il élimine le risque de fuite de données résiduelles sur le disque. Dans un lab, le Thin Provisioning est acceptable pour économiser de l'espace, mais il faut en comprendre les implications.

A retenir
Un Datastore est l'abstraction de stockage dans vSphere. Les VMs y stockent leurs fichiers. En cybersécurité, le stockage doit être contrôlé : chiffrement au repos (VM Encryption avec vCenter + KMS), accès restreint aux datastores, surveillance de l'espace disponible.

B.5 — Sécurité vSphere / vCenter

La sécurisation d'un environnement vSphere repose sur plusieurs axes complémentaires :

RBAC — Roles-Based Access Control

vCenter intègre un système de permissions à trois éléments : Utilisateur (ou groupe), Rôle (ensemble de privilèges), Objet (Datacenter, Cluster, VM, folder…). Un rôle est assigné à un utilisateur sur un objet spécifique, avec ou sans propagation aux enfants. vCenter fournit des rôles prédéfinis (Administrator, Read-Only, VM Power User, etc.) et permet d'en créer des personnalisés.

Figure B.5a — Modèle de permission vCenter (triplet)

Surface d'attaque et durcissement

SurfaceRisqueMesure de durcissement

Comptes par défautMot de passe admin connu ou faibleChanger immédiatement, principe du moindre privilège

A retenir
La sécurité de vCenter/vSphere repose sur le moindre privilège (RBAC fin), la segmentation réseau (management isolé), la journalisation (syslog centralisé), la synchronisation temporelle (NTP) et la gestion des certificats. Ce sont des fondamentaux applicables à toute infrastructure.
Micro-TP (5 min) — Audit rapide ESXi

Sur votre ESXi, vérifiez : le service SSH est-il activé ? (Menu Host → Manage → Services). Si oui, notez-le comme un point à durcir. Vérifiez aussi la configuration NTP (Manage → System → Time & Date).

B.6 — Modèle objet Proxmox « Datacenter »

Dans Proxmox VE, la vue Datacenter est le niveau le plus élevé de l'interface d'administration. Même avec un seul nœud, cette vue centralise la configuration des éléments partagés : utilisateurs, rôles, ACL, stockage, firewall, sauvegardes planifiées et réplication.

Micro-TP (5 min) — Exploration Datacenter

Connectez-vous à votre Proxmox (https://<IP_Proxmox>:8006). Cliquez sur « Datacenter » dans l'arbre de gauche. Parcourez chaque sous-menu (Search, Summary, Cluster, Options, Storage, Backup, Replication, Permissions, Firewall). Notez le nombre de sous-menus disponibles.

B.7 — Réseau Proxmox

Proxmox utilise le modèle réseau natif de Linux. Par défaut, un bridge Linux nommé vmbr0 est créé à l'installation, relié à l'interface réseau physique. Les VMs et conteneurs se connectent à ce bridge. C'est l'équivalent fonctionnel d'un vSwitch dans vSphere.

Figure B.7 — Réseau Proxmox : bridge Linux vmbr0
Sécurité — Isolation réseau Proxmox

Par défaut, le trafic management et le trafic VM partagent le même bridge vmbr0. Pour durcir l'environnement, créez un second bridge vmbr1 (sans adresse IP, sans uplink physique) dédié aux VMs de lab. Cela isole les VMs du réseau de management — principe de segmentation fondamental.

A retenir
Proxmox utilise des bridges Linux au lieu de vSwitches propriétaires. Le concept est identique : connecter des VMs à un réseau virtuel. Pour le VLAN tagging, Proxmox supporte la configuration native Linux (bridge-vlan-aware). La segmentation management / VMs passe par la création de bridges séparés.

B.8 — Stockage Proxmox

Proxmox gère le stockage via deux types principaux par défaut :

StockageTypeContenuUsage
localDirectory (/var/lib/vz)ISO, templates CT, backups, snippetsStockage de fichiers (non-VM)
local-lvmLVM-ThinDisques VM (images), rootfs CTStockage des machines virtuelles

Cette séparation est importante : les ISOs vont dans local, les disques VM dans local-lvm. On peut ajouter d'autres stockages (NFS, CIFS, Ceph, ZFS) via la vue Datacenter → Storage.

Info — LVM-Thin

LVM-Thin (Thin Provisioning) est l'équivalent Proxmox du Thin Provisioning VMware : l'espace disque n'est alloué que lorsqu'il est réellement écrit. Cela permet de surprovisonner le stockage, mais nécessite une surveillance de l'espace réel utilisé.

A retenir
Proxmox sépare les stockages par type de contenu : ISO/backups dans local, disques VM dans local-lvm. Cette séparation facilite le contrôle d'accès et la gestion de l'espace.

B.9 — Sécurité Proxmox

Realms d'authentification

Proxmox supporte plusieurs sources d'authentification (realms) :

RealmDescriptionUsage typique
PAMUtilisateurs Linux locaux du nœud ProxmoxCompte root@pam (administrateur par défaut)
PVEBase d'utilisateurs interne à Proxmox (indépendante de Linux)Comptes étudiants, opérateurs — sans accès shell Linux
LDAP / ADAuthentification externe (Active Directory, OpenLDAP)Environnement d'entreprise, SSO

RBAC Proxmox

Le modèle RBAC de Proxmox est similaire à celui de vCenter : on attribue un rôle (ensemble de privilèges) à un utilisateur (ou groupe) sur un chemin d'objet (ex: /pool/CYBER-LAB, /vms/100, / pour tout). Proxmox fournit des rôles prédéfinis : Administrator, PVEVMAdmin, PVEVMUser, PVEAuditor, etc.

API Tokens

Proxmox permet de créer des tokens API pour l'automatisation, avec des privilèges potentiellement restreints par rapport au compte parent. Chaque token possède un identifiant et un secret, et peut être limité à un sous-ensemble des permissions du compte.

Firewall intégré multi-niveaux

Le pare-feu Proxmox opère à trois niveaux indépendants, chacun avec ses propres règles :

NiveauPortéeConfiguration
DatacenterRègles globales appliquées à tous les nœudsDatacenter → Firewall
NodeRègles spécifiques à un nœud (trafic entrant vers le nœud)Node → Firewall
VM / CTRègles par machine virtuelle ou conteneurVM → Firewall
Sécurité — Firewall Proxmox : prudence en lab

Le firewall Proxmox est désactivé par défaut. L'activer sans avoir configuré de règle autorisant le management (port 8006, SSH 22) vous bloquera l'accès à votre nœud. Ajoutez toujours les règles d'accès management AVANT d'activer le firewall.

A retenir
Proxmox offre un écosystème de sécurité complet : realms multiples (PAM/PVE/LDAP), RBAC fin (utilisateurs/groupes/rôles/ACL sur chemins), API tokens pour l'automatisation sécurisée, firewall multi-niveaux intégré, et journaux consultables dans l'interface.
Micro-TP (5 min) — Permissions Proxmox

Dans Proxmox, allez dans Datacenter → Permissions → Roles. Listez les rôles prédéfinis et notez les privilèges du rôle PVEVMUser. Comparez avec PVEAdmin. Quelle est la différence principale ?

C — TP guidé : vCenter Server

Info — Objectif de cette section

Déployer vCenter Server Appliance (VCSA), connecter votre hôte ESXi, construire une hiérarchie Datacenter/Cluster/Folders, créer une VM, manipuler snapshots, configurer le RBAC et vérifier les logs. Chaque étape intègre un volet cybersécurité.

Étape VC-1 — Déployer vCenter Server Appliance (VCSA)
Action

Le déploiement de VCSA se fait depuis un poste Windows/Linux/Mac disposant de l'ISO VCSA monté :

  • Montez l'ISO VMware-VCSA-all-*.iso sur votre machine hôte (Windows : double-clic sur l'ISO)
  • Naviguez vers vcsa-ui-installer/win32/ (ou lin64/) et lancez installer.exe
  • Choisissez Install → sélectionnez le type Embedded Platform Services Controller
  • Cible ESXi : entrez l'IP de votre hôte ESXi et les identifiants root
  • Nom de la VM VCSA : vcsa-cyber-lab
  • Taille : Tiny (2 vCPU, 12 Go RAM) — c'est le minimum, exigeant en nested
  • Datastore : sélectionnez le datastore local de l'ESXi, cochez Thin Disk Mode
  • Réseau : Port Group VM Network (ou votre port group de management)
  • Configurez : IP statique pour VCSA, masque, passerelle, DNS, NTP, mot de passe root VCSA
  • Phase 2 (configuration SSO) : domaine SSO vsphere.local, mot de passe admin SSO
Info — Explication

VCSA est une appliance Linux (Photon OS) qui embarque vCenter Server, le SSO (Single Sign-On) et la base de données PostgreSQL. Le déploiement en deux phases installe d'abord la VM, puis configure les services internes. Avec 12 Go de RAM, c'est un composant gourmand — d'où la nécessité d'avoir suffisamment de RAM sur l'hôte Workstation.

Vérification

Après déploiement (30–60 minutes), accédez à https://<IP_VCSA>:443 dans un navigateur. Vous devez voir la page de connexion vSphere Client. Connectez-vous avec administrator@vsphere.local et le mot de passe SSO défini. Capture exigée : page d'accueil vSphere Client.

Sécurité

Le mot de passe SSO doit être robuste (12+ caractères, complexité). Notez que VCSA utilise un certificat auto-signé par défaut — le navigateur affichera un avertissement. En production, remplacez ce certificat. Limitez l'accès réseau au port 443 de VCSA aux seuls postes d'administration.

Erreur fréquente — VCSA inaccessible après déploiement

Symptôme : La page https://IP_VCSA ne répond pas. Causes possibles : les services VCSA mettent 10–15 min à démarrer après le premier boot ; le DNS ne résout pas le FQDN configuré ; l'IP est en conflit. Solution : attendez, vérifiez la console VCSA dans ESXi (touche F2 pour voir l'IP), vérifiez les entrées DNS/hosts.

Étape VC-2 — Se connecter à vCenter et comprendre l'interface
Action

Ouvrez https://<IP_VCSA> dans un navigateur. Connectez-vous avec administrator@vsphere.local. Explorez l'interface :

  • Menu hamburger (en haut à gauche) : accès aux vues (Hosts & Clusters, VMs & Templates, Storage, Networking)
  • Inventaire (panneau gauche) : arborescence des objets
  • Panneau central : détails de l'objet sélectionné (Summary, Monitor, Configure, Permissions…)
  • Recent Tasks (en bas) : historique des opérations
Vérification

L'inventaire doit être vide (aucun datacenter encore créé). Le menu « Hosts and Clusters » ne montre que le nœud vCenter lui-même. Capture : vue d'inventaire vide.

Étape VC-3 — Créer un Datacenter
Action

Dans la vue « Hosts and Clusters » : clic droit sur l'icône vCenter (racine) → New Datacenter. Nommez-le CYBER-DC. Validez.

Info — Explication

Le Datacenter est le conteneur logique de plus haut niveau. Il regroupe clusters, hôtes, réseaux et datastores. Dans un environnement réel, on peut avoir plusieurs Datacenters (un par site géographique, par exemple). Les permissions appliquées au Datacenter se propagent à tous les objets enfants.

Vérification

Le Datacenter CYBER-DC apparaît dans l'inventaire. En le sélectionnant, l'onglet « Summary » affiche 0 cluster, 0 host. Capture exigée.

Étape VC-4 — Créer un Cluster
Action

Clic droit sur CYBER-DC → New Cluster. Nommez-le CYBER-CLUSTER. Options :

  • vSphere DRS : laisser désactivé (un seul hôte, pas de migration possible)
  • vSphere HA : laisser désactivé (nécessite 2+ hôtes)

Validez la création.

Info — Pourquoi un cluster à 1 hôte ?

Même avec un seul hôte, le cluster fournit un cadre organisationnel. Il sera prêt à accueillir un second hôte si vous étendez le lab. De plus, certaines fonctionnalités (Resource Pools, règles d'affinité) ne sont disponibles qu'au sein d'un cluster.

Vérification

Le cluster CYBER-CLUSTER apparaît sous CYBER-DC dans l'inventaire, avec un triangle d'avertissement (aucun hôte membre). Normal à ce stade.

Étape VC-5 — Ajouter l'hôte ESXi au cluster
Action

Clic droit sur CYBER-CLUSTER → Add Host. Entrez :

  • IP ou FQDN de votre ESXi : 192.168.x.10 (adaptez)
  • Identifiant : root
  • Mot de passe : celui défini lors de l'installation ESXi

Acceptez l'alerte de certificat (thumbprint SHA-1). Validez l'assistant.

Info — Explication

Lorsqu'un hôte rejoint vCenter, il délègue une partie de sa gestion à vCenter (permissions, réseau distribué, etc.). L'hôte reste accessible directement (Host Client) mais les modifications centralisées passent désormais par vCenter. C'est un changement fondamental de paradigme d'administration.

Vérification

L'hôte ESXi apparaît sous CYBER-CLUSTER avec un statut « Connected » (icône verte). L'onglet « Summary » de l'hôte affiche CPU, RAM, datastores. Les VMs éventuellement déjà présentes sur l'hôte sont visibles dans l'inventaire. Capture exigée.

Sécurité — Vérification du thumbprint

L'alerte de certificat lors de l'ajout d'un hôte est une protection contre le MITM. En production, vérifiez le thumbprint SHA-256 de l'hôte via la console directe avant de l'accepter dans vCenter.

Erreur fréquente — ESXi non ajoutable

Symptôme : erreur « Cannot contact the specified host » ou timeout. Causes : IP incorrecte, ESXi non démarré, pare-feu hôte bloquant vCenter, DNS ne résolvant pas. Solution : vérifiez la connectivité (ping depuis VCSA), vérifiez que le port 443 de l'ESXi est accessible, vérifiez que le firewall ESXi n'est pas restrictif.

Étape VC-6 — Organiser l'inventaire (Folders et nommage)
Action

Passez en vue « VMs and Templates » (menu hamburger). Clic droit sur CYBER-DC → New Folder → New VM and Template Folder :

  • Créez un folder Lab-VMs (contiendra les VMs de travail)
  • Créez un folder Templates (contiendra les templates/golden images)
  • Créez un folder Quarantine (pour les VMs suspectes à isoler)
Info — Conventions de nommage

Adoptez une convention cohérente dès le départ : CYBER-DC, CYBER-CLUSTER, Lab-VMs. Les noms doivent refléter la fonction et être lisibles. Cela facilite l'application des permissions par folder et l'audit de l'environnement.

Vérification

Étape PX-2 — Cluster Proxmox (concept et optionnel)
Info — Cluster à 1 nœud vs multi-nœuds

Un cluster Proxmox permet de fédérer plusieurs nœuds sous une même vue Datacenter. Avec un seul nœud, le clustering n'est pas strictement nécessaire, mais la vue Datacenter fonctionne déjà comme un mini-cluster. En production, le cluster Proxmox utilise Corosync pour la communication inter-nœuds et permet la migration de VMs (live migration avec stockage partagé).

Dans ce TP, nous travaillons avec un seul nœud. La création d'un cluster à 1 nœud est possible (pvecm create CYBER-CLUSTER) mais optionnelle — elle ne change pas les fonctionnalités accessibles dans la vue Datacenter pour notre lab.

Discussion orientée cybersécurité

Sécurité — Recommandations par contexte

Lab cybersécurité / Pentest lab : Proxmox est avantageux — gratuit, backup intégré, firewall intégré, déploiement léger. Les snapshots et la restauration rapide sont essentiels pour les tests offensifs.

PME : Proxmox offre un excellent rapport fonctionnalités/coût. vCenter est pertinent si l'entreprise utilise déjà l'écosystème VMware et dispose du budget licences.

SOC / Entreprise : vCenter avec NSX apporte une micro-segmentation avancée et une intégration profonde avec les outils de sécurité VMware (Carbon Black, etc.). Proxmox peut alimenter un SOC via ses API et syslog.

Audit / Conformité : vCenter dispose d'outils de conformité matures (vSphere Security Configuration Guide). Proxmox nécessite plus de configuration manuelle mais est tout aussi auditable via logs et API.

Conclusion
Ce TP vous a permis de passer de l'administration d'un hôte isolé à une gestion centralisée sur deux plateformes majeures de virtualisation. Que ce soit avec vCenter (écosystème VMware propriétaire) ou la vue Datacenter de Proxmox (écosystème open source), les concepts fondamentaux sont les mêmes : hiérarchie d'objets, contrôle d'accès, segmentation réseau, journalisation et résilience (snapshots/backups). Ces compétences sont directement applicables dans tout contexte professionnel de cybersécurité, que ce soit pour administrer un SOC, construire un lab de pentest ou auditer une infrastructure virtualisée.
