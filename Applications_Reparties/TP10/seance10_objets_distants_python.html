De RPC à l'Invocation d'Objets Distants

A. Rappel : qu'est-ce que RPC ?

En RPC (Remote Procedure Call), un client invoque une fonction hébergée sur un serveur distant, comme s'il s'agissait d'une fonction locale. Le client ne manipule aucun état côté serveur entre les appels. Chaque appel est autonome et indépendant.

💡 Analogie RPC

Appeler un RPC ressemble à commander une pizza par téléphone : vous dites exactement ce que vous voulez (arguments), le pizzaiolo exécute (procédure distante), vous recevez le résultat (valeur de retour). Vous ne gérez pas le four, la pâte, ni l'état interne de la pizzeria.

B. L'invocation d'objet distant — un pas au-delà

Avec l'invocation d'objet distant, le serveur expose non plus une fonction isolée, mais un objet entier avec plusieurs méthodes, un état interne, et une identité persistante. Le client obtient une référence à cet objet et peut appeler plusieurs méthodes dessus.

Fig. 1 — RPC (appel de procédure) vs Invocation d'objet distant
⚠️ Piège Classique : croire que l'objet distant est local

Lorsqu'un client utilise un proxy, il écrit doc.get_content("doc1") exactement comme si doc était un objet local. Mais derrière, chaque appel traverse le réseau, subit une latence, peut échouer, et implique une sérialisation/désérialisation des données.

  • Ne pas appeler des méthodes distantes dans des boucles serrées (N+1 appels réseau)
  • Toujours traiter les exceptions réseau comme une possibilité réelle
  • Ne pas supposer que l'état distant est synchronisé avec la logique locale

C. La notion de référence distante

En Java RMI, un stub est généré automatiquement. En Python avec Pyro5, le proxy joue ce rôle : c'est un objet local qui intercepte les appels de méthodes et les achemine vers l'objet réel sur le serveur distant, via le réseau.

💡 Exemple Concret

Imaginez un annuaire téléphonique d'entreprise. Vous cherchez "Service Comptabilité", vous obtenez un numéro (référence distante). Quand vous appelez ce numéro, vous parlez au vrai service comptabilité (objet distant). Votre téléphone (proxy) gère la connexion de manière transparente.

C. Le rôle du registre (Name Server)

Le name server est un processus indépendant qui joue le rôle d'annuaire des objets distants. Il permet le découplage entre client et serveur :

  • Le serveur enregistre son objet sous un nom logique : "example.calculator"
  • Le client récupère l'URI de l'objet en cherchant par ce nom logique
  • Si le serveur change d'adresse IP, seul le name server est mis à jour — le client ne change pas
  • Possibilité d'enregistrer plusieurs objets différents sous des noms distincts
✅ À Retenir — Name Server = Découplage Architectural

Le client ne connaît jamais directement l'adresse IP ni le port du serveur d'objets. Il connaît seulement l'adresse du name server et le nom logique de l'objet. C'est exactement le même principe que DNS pour les noms de domaine.

Pyro5 — Technologie Python Principale

Pyro5 (Python Remote Objects) est la bibliothèque Python qui incarne le plus fidèlement l'idée de RMI : exposer des objets Python entiers à travers le réseau, avec name server, proxy transparent et gestion des exceptions distantes.

06

Publication et Découverte d'Objets Distants

A. Comment un serveur publie un objet

La publication d'un objet implique trois étapes :

  • Définir la classe décorée @Pyro5.api.expose pour rendre ses méthodes accessibles
  • Créer un daemon : processus serveur qui héberge et gère la communication réseau
  • Enregistrer l'objet dans le name server sous un nom logique
✅ À Retenir — Exposition Minimale

Par défaut, aucune méthode n'est accessible à distance. Seules les méthodes marquées explicitement @Pyro5.api.expose sont publiées. C'est une politique de sécurité fondamentale : liste blanche, pas liste noire.

B. Comment le client découvre l'objet

  • Le client se connecte au name server (adresse connue ou par broadcast)
  • Il demande l'URI de l'objet via son nom logique
  • Il crée un proxy Pyro5.api.Proxy(uri)
  • Il appelle les méthodes sur le proxy — comme sur un objet local
💡 Le nom logique — pourquoi c'est important

Le nom logique ("bank.documents.service") dissocie l'identité du service de son adresse physique. Dans une architecture distribuée avec plusieurs environnements (dev/staging/prod), on change l'URI enregistrée dans le name server — mais les clients ne changent pas. C'est le principe du découplage par nommage.

Limites et Risques du Modèle Objet Distant

  • Latence cachée : obj.method() ressemble à un appel local, mais traverse le réseau. La syntaxe masque la réalité de la distribution.
  • Erreurs réseau inattendues : coupure, timeout, rejet de connexion — sur des appels qui "ressemblent" à des appels locaux.
  • Couplage fort : si l'interface de l'objet change, les clients doivent être mis à jour.
  • Exposition accidentelle de méthodes : si la décoration est mal contrôlée.
  • Sérialisation implicite : tout paramètre et tout résultat est automatiquement sérialisé — des structures complexes ou des objets inattendus peuvent passer.
  • Exceptions distantes : une exception côté serveur est re-levée côté client, potentiellement avec des informations internes sensibles dans le traceback.
  • Problème N+1 : appeler une méthode distante dans une boucle génère N requêtes réseau — catastrophique pour les performances.
⚠️ Piège Classique — L'illusion de la localité

Un objet distant ressemble à un objet local. Cette ressemblance est précisément ce qui le rend dangereux : le développeur oublie facilement les contraintes réseau, les erreurs de sérialisation, et les problèmes de disponibilité. Toujours traiter un appel distant comme une opération pouvant échouer.

B. Mesures de protection

🔐 Point Cybersécurité — Principes fondamentaux
  • Exposition minimale : ne décorer que ce qui doit être accessible — principe du moindre privilège
  • Validation systématique : type, longueur, format — côté serveur, jamais côté client seul
  • Sérialisation sûre : préférer Serpent à pickle en Pyro5 ; jamais de structures arbitraires
  • Erreurs discrètes : message générique au client, trace complète dans les logs internes
  • Timeouts : tout appel distant doit avoir une durée maximale configurable
  • Journalisation contrôlée : loguer les appels, les erreurs, sans exposer de secrets
  • Séparation dev/test/prod : ne jamais utiliser un name server de prod en dev
  • Audit régulier : inventorier les objets enregistrés dans le name server

  • État interne géré côté serveur (cache, compteurs d'accès)
  • Consommé par plusieurs clients Python internes

  • Pyro5/objets distants : si tous les acteurs sont en Python et la richesse objet est nécessaire

Live Coding Python — Démonstrations Guidées

Démo 1 — Publication d'un objet distant simple (CalculatorService)

Ce premier exemple illustre les briques minimales : objet exposé, daemon, name server, client proxy.

server_calculator.py
import Pyro5.api

@Pyro5.api.expose          # Rend les méthodes accessibles à distance
class CalculatorService:
    """Service de calcul exposé comme objet distant."""

    def add(self, a: float, b: float) -> float:
        # Valider les types — ne jamais faire confiance au client
        if not (isinstance(a, (int, float)) and isinstance(b, (int, float))):
            raise ValueError("Les arguments doivent être des nombres")
        return a + b

    def subtract(self, a: float, b: float) -> float:
        if not (isinstance(a, (int, float)) and isinstance(b, (int, float))):
            raise ValueError("Les arguments doivent être des nombres")
        return a - b

    def _internal_reset(self):
        # Méthode INTERNE — PAS de @expose ici = non accessible à distance
        pass


def main():
    # Créer le daemon (serveur d'objets)
    with Pyro5.api.Daemon() as daemon:
        # Localiser le name server
        ns = Pyro5.api.locate_ns()

        # Enregistrer l'objet dans le daemon → obtenir son URI unique
        uri = daemon.register(CalculatorService)

        # Publier l'URI dans le name server sous un nom logique
        ns.register("example.calculator", uri)

        print(f"CalculatorService prêt. URI: {uri}")
        daemon.requestLoop()    # Boucle d'attente infinie (Ctrl+C pour arrêter)


if __name__ == "__main__":
    main()
client_calculator.py
import Pyro5.api

def main():
    # Localiser le name server
    ns = Pyro5.api.locate_ns()

    # Récupérer l'URI du service par son nom logique
    uri = ns.lookup("example.calculator")

    # Créer le proxy — l'objet "local" qui représente le CalculatorService distant
    with Pyro5.api.Proxy(uri) as calc:
        result_add = calc.add(10, 5)       # → appel réseau transparent
        result_sub = calc.subtract(10, 3)  # → appel réseau transparent
        print(f"10 + 5 = {result_add}")
        print(f"10 - 3 = {result_sub}")


if __name__ == "__main__":
    main()
✅ Points clés de cette démo
  • @Pyro5.api.expose est la liste blanche de ce qui est accessible — sans ce décorateur, la méthode est invisible à distance
  • Daemon = le serveur réseau qui héberge l'objet
  • locate_ns() cherche le name server sur le réseau local
  • Proxy(uri) = l'objet local transparente qui délègue tout au serveur
  • La méthode _internal_reset est inaccessible à distance car non décorée

Démo 2 — DocumentService orienté objet avec validation

server_documents.py
import Pyro5.api
import logging
import re

logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(message)s'
)
logger = logging.getLogger(__name__)

# Données simulées côté serveur
_DOCUMENTS = {
    "doc_001": "Rapport annuel 2024 — données confidentielles",
    "doc_002": "Politique de sécurité — version 3.2",
    "doc_003": "Guide d'utilisation — accès public",
}

@Pyro5.api.expose
class DocumentService:
    """Service de gestion de documents — équivalent RMI Python."""

    def list_documents(self) -> list:
        """Retourne la liste des IDs de documents disponibles."""
        logger.info("Appel list_documents()")
        return list(_DOCUMENTS.keys())

    def get_document_content(self, doc_id: str) -> str:
        """Retourne le contenu d'un document après validation stricte."""
        # 1. Validation du type
        if not isinstance(doc_id, str):
            logger.warning(f"Type invalide reçu pour doc_id: {type(doc_id)}")
            raise TypeError("Identifiant invalide")  # message générique

        # 2. Validation du format (alphanumérique + underscore uniquement)
        if not re.match(r'^[a-zA-Z0-9_]{1,32}$', doc_id):
            logger.warning(f"Format doc_id non conforme: {doc_id!r}")
            raise ValueError("Identifiant invalide")  # message générique

        # 3. Vérification existence
        if doc_id not in _DOCUMENTS:
            logger.info(f"Document non trouvé: {doc_id}")
            raise KeyError("Document introuvable")

        logger.info(f"Document servi: {doc_id}")
        return _DOCUMENTS[doc_id]

    def _reload_index(self):
        # Méthode INTERNE — PAS d'@expose → inaccessible à distance
        pass


def main():
    with Pyro5.api.Daemon() as daemon:
        ns = Pyro5.api.locate_ns()
        uri = daemon.register(DocumentService)
        ns.register("bank.documents.service", uri)
        print(f"DocumentService prêt — URI: {uri}")
        daemon.requestLoop()

if __name__ == "__main__":
    main()

Démo 3 — Erreurs distantes sécurisées

Cette démo compare deux approches : une naïve qui fuit trop d'informations, et une défensive qui protège l'information interne.

Fig. 4 — Exception distante : message générique au client, log complet en interne
Version ❌ NAÏVE — fuite d'informations
@Pyro5.api.expose
class NaiveService:
    def get_data(self, key):
        # ❌ L'exception raw est propagée au client avec TOUS les détails
        # Traceback inclut: chemin fichier, variables locales, stack complet
        data = self._load_from_db(key)  # peut lever FileNotFoundError
        return data                        # ← info système exposée au client
Version ✅ DÉFENSIVE — erreurs sûres
import logging
logger = logging.getLogger(__name__)

@Pyro5.api.expose
class SecureService:
    def get_data(self, key: str) -> dict:
        # 1. Validation entrée
        if not isinstance(key, str) or not key.isalnum():
            logger.warning(f"Clé invalide reçue: {key!r}")
            raise ValueError("Paramètre invalide")   # ← générique

        try:
            data = self._load_from_db(key)
            return data

        except KeyError:
            logger.info(f"Ressource non trouvée: {key}")
            raise KeyError("Ressource introuvable")  # ← générique

        except Exception as e:
            # Log complet en interne — jamais visible du client
            logger.error(f"Erreur interne get_data({key!r}): {e}", exc_info=True)
            # Message générique vers le client — pas de fuite
            raise RuntimeError("Erreur de service. Contactez l'administrateur.")
🔐 Point Cybersécurité — Deux audiences, deux messages

Client → reçoit un message générique, actionnable, sans information système. Équipe technique → voit les logs complets avec contexte, fichier, ligne, variable. Cette séparation est une pratique fondamentale de sécurité (information disclosure prevention). Ne jamais laisser un traceback Python brut atteindre un client externe.

TP10.3 — Validation Stricte des Entrées et Erreurs Sûres (20 min)

Règles de validation à implémenter

TP10.4 — Surface d'Attaque et Sérialisation Sûre (20 min)

🎯 Objectif

Analyser ce qui traverse le réseau et minimiser la surface d'attaque liée à la sérialisation des objets distants.

Checklist — Safe Remote Object Calls

  • Sérialiseur Serpent (défaut Pyro5) — jamais pickle en production
  • Paramètres : types primitifs uniquement (str, int, float, bool, list, dict de primitifs)
  • Retours : types simples — pas d'objets Python complexes ou de classes custom
  • Aucun objet de fichier, socket, ou connexion DB transmis comme paramètre ou retour
  • Aucune structure récursive profonde ou graphe d'objets complexe
  • Valider le type et la taille de chaque paramètre avant traitement
  • Journaliser tout paramètre rejeté avec contexte
  • Pas de lambda, callable, ou code transmis comme argument

Checklist — Remote Object Security Baseline

  • Exposition minimale : inventaire de toutes les méthodes décorées @expose
  • Pas de données sensibles dans les attributs publics : jamais de secrets dans les instances exposées
  • Validation type et format de tous les paramètres entrants
  • Gestion d'exceptions systématique : aucune exception brute propagée au client
  • Journalisation contrôlée : logs détaillés en interne, jamais de secrets dans les logs
  • Authentification minimale : token ou mécanisme équivalent sur les méthodes sensibles
  • Timeouts configurés côté client pour chaque appel distant
  • Sérialiseur sûr : Serpent (défaut Pyro5), jamais pickle sans analyse
  • Séparation dev/prod : name server de dev jamais accessible en production
  • Audit régulier : lister les objets enregistrés dans le name server (ns.list())

Mini-Quiz — 10 Questions + Corrigé

Question 01
Quelle est la différence principale entre RPC et l'invocation d'objet distant ?
A — RPC est plus rapide car il n'utilise pas de réseau
B — RPC appelle une fonction isolée ; l'invocation d'objet distant manipule un objet avec état et plusieurs méthodes
C — L'invocation d'objet distant ne nécessite pas de sérialisation
D — RPC est uniquement utilisé en Java
✅ Réponse B. RPC est stateless et cible une procédure unique. L'invocation d'objet distant expose une instance entière côté serveur, avec plusieurs méthodes et un état interne persistant.
Question 02
Dans Pyro5, que fait le décorateur @Pyro5.api.expose ?
A — Il chiffre les données transmises sur le réseau
B — Il définit la méthode comme méthode de classe Python
C — Il rend la méthode accessible aux clients distants (liste blanche)
D — Il enregistre automatiquement l'objet dans le name server
✅ Réponse C. Sans ce décorateur, la méthode est complètement invisible aux clients. C'est une politique de liste blanche : seul ce qui est explicitement marqué est exposé.
Question 03
Quel est le rôle du proxy dans le modèle d'invocation d'objet distant ?
A — Stocker une copie locale de l'objet distant pour éviter les appels réseau
B — Intercepter les appels de méthodes locaux et les acheminer vers l'objet réel sur le serveur
C — Gérer l'authentification des clients
D — Sérialiser les données dans le format XML
✅ Réponse B. Le proxy est un objet "fantôme" côté client. Il ressemble à l'objet réel, mais chaque appel de méthode est transmis au serveur via le réseau, de façon transparente.
Question 04
Pourquoi le name server améliore-t-il la maintenabilité d'une architecture distribuée ?
A — Il compresse les données pour accélérer les transferts
B — Il authentifie les clients avant de leur donner accès
C — Le client connaît seulement le nom logique — si le serveur change d'adresse, seul le name server est mis à jour
D — Il répartit automatiquement la charge entre plusieurs serveurs
✅ Réponse C. Le nom logique découple client et serveur. C'est le même principe que DNS : vous tapez "google.com", pas une adresse IP — si l'adresse change, vous ne le savez pas.
Question 05
Quel est le risque principal d'utiliser pickle comme sérialiseur dans Pyro5 ?
A — Les données sont trop volumineuses et ralentissent le réseau
B — Un client malveillant peut envoyer un objet pickle qui exécute du code arbitraire sur le serveur (RCE)
C — Pickle ne supporte pas les dictionnaires Python
D — Pickle est incompatible avec Python 3
✅ Réponse B. Pickle est deserialization by execution : désérialiser un pickle malveillant peut exécuter du code arbitraire. C'est pourquoi Pyro5 utilise Serpent par défaut, qui est bien plus restrictif.
Question 06
Quelle est la bonne pratique concernant les messages d'erreur dans un service distant ?
A — Toujours retourner le traceback complet pour faciliter le débogage
B — Ne jamais lever d'exception — retourner None en cas d'erreur
C — Message générique au client, log complet avec contexte en interne
D — Envoyer l'erreur par email à l'administrateur
✅ Réponse C. Deux audiences, deux niveaux d'information. Le client voit "Erreur de service" ; l'équipe interne voit la ligne, le fichier, la variable, le contexte complet. Cette séparation prévient la fuite d'informations architecturales.
Question 07
Une méthode _delete_all_documents() doit-elle être exposée à distance ? Pourquoi ?
A — Oui, si le client en a besoin
B — Oui, après authentification uniquement
C — Non — une opération aussi critique ne doit jamais être accessible via un objet distant standard ; elle relève d'un channel d'administration séparé
D — Oui, mais avec un mot de passe en paramètre
✅ Réponse C. Le principe du moindre privilège s'applique strictement. Les opérations destructrices doivent passer par des canaux d'administration dédiés, isolés du service métier normal.
Question 08
Que signifie "N+1 appels réseau" dans le contexte des objets distants ?
A — Appeler une méthode distante dans une boucle génère N requêtes réseau — chacune avec sa latence, ce qui est très coûteux
B — Le name server doit être contacté N fois avant chaque appel
C — Le proxy nécessite N+1 connexions TCP simultanées
D — Pyro5 envoie N copies de chaque message pour la fiabilité
✅ Réponse A. Si vous avez 100 documents et appelez get_content(id) pour chacun dans une boucle, vous générez 100 appels réseau. La solution : concevoir des méthodes qui retournent des lots (batch) de données.
Question 09
Quel composant Pyro5 correspond au concept de "registre" dans Java RMI ?
A — Pyro5.api.Proxy
B — Pyro5.api.Daemon
C — Pyro5.nameserver — le name server
D — @Pyro5.api.expose
✅ Réponse C. Le Pyro5.nameserver est l'équivalent du rmiregistry Java : un processus indépendant qui maintient un annuaire nom logique → URI de l'objet distant.
Question 10
Dans quel cas préférer gRPC à Pyro5 pour un service distribué ?
A — Quand tous les clients sont écrits en Python
B — Quand on veut exposer un objet Python avec état côté serveur
C — Quand les performances sont critiques, le contrat API doit être strict (Protobuf) et les clients peuvent être dans différents langages
D — Quand on préfère éviter la sérialisation des données
✅ Réponse C. gRPC brille pour les systèmes hétérogènes multi-langages avec des exigences de performance. Pyro5 est idéal pour des systèmes 100% Python nécessitant la richesse du modèle objet. Ni l'un ni l'autre n'est universellement supérieur.
